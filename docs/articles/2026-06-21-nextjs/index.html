Motion（旧Framer Motion）を Next.js App Router で — 'use client'境界・template.tsxでenter・LazyMotionで軽量化

Reactで最も使われる宣言的アニメライブラリ Framer Motion は、2024年末に「Motion」へ改名され、Reactエントリは motion/react に統合された。App Router では motion.* がClient専用なため 'use client' 必須、新ルートを先に描く特性でページexitが効きにくいなど固有の落とし穴がある。enterは template.tsx、exitは View Transitions と役割分担し、LazyMotion でバンドルを削る組み込み方を整理する。

なぜ今これが話題か

Reactで最も使われる宣言的アニメライブラリ Framer Motion は、2024年末に 「Motion」へ改名された。Reactエントリは motion/react、バニラJSは motion に統合。framer-motion は互換エイリアスとして動くが、新規は motion/react が推奨だ。App Router での正しい組み込みを押さえておきたい。

何が新しい/重要か

強みは3点。initial/animate/exit の宣言だけでマウント/アンマウント時のexitまで書ける、layout/layoutId によるFLIPベースのレイアウト/共有要素アニメを内部で自動処理、transform/opacityは WAAPI（ハイブリッドエンジン）でGPU合成へオフロードしメインスレッドを空ける。一方で App Router 特有の落とし穴があるため、組み込み方の理解が要る。

App Router 特有の落とし穴

motion.* はフック/Contextを使うClient専用なので 'use client' が必須。ページはServerのまま、動く部分だけ小さなClientラッパーに切り出すのが定石だ。さらに App Router は新ルートを描画してから旧セグメントを外すため、ページのexitが効きにくい。enterだけなら template.tsx（遷移ごとに再マウント）に motion.div を置けば毎回 initial→animate が走る。exitも欲しければ View Transitions API との併用が無難だ。

仕組みのポイント

バンドルが気になるなら LazyMotion + m + domAnimation（軽量）/domMax（layout等フル）で機能を遅延ロードし初期JSを削れる。アクセシビリティは <MotionConfig reducedMotion="user"> を上位に置くと、配下のtransform/opacity系の動きをユーザ設定に従って一括抑制できる（動きは消してもフェードや状態変化は残す）。

自分が試すなら

  • 共有要素タブのインジケータを layoutId="indicator" で滑らかに移動させる。
  • enter演出は template.tsx、ページ遷移は View Transitions と役割分担する。
  • LazyMotion + m で初期JS削減の効果を計測する。

注意点

初回ペイントで initial がレイアウトをずらすとCLSになり得る。位置に影響しない opacity/transform を使うか、初回は initial={false}。AnimatePresence 直下の子には一意の key が必須。width/height などレイアウト誘発プロパティはJS駆動でコスト高なので scale/transform に寄せる。

ひとことまとめ

App Router では「動く所だけ 'use client'、enterは template.tsx、exitは View Transitions」が素直。まずは whileInView のReveal一つから入れて体感するのが手軽だ。
