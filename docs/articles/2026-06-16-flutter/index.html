Flutter 3.44 の構造改革 — Material/Cupertinoの独立化とSwiftPM既定化が意味すること

Material/Cupertinoが本体から分離してpub.dev配布へ。iOS/macOSの依存管理はSwiftPMが既定に昇格しCocoaPods終焉へ。Riverpod 3.3とImpellerの現在地も合わせて整理する。

📅 2026年6月16日 ⏱ 約4分 🔗 出典 4 件

なぜ今これが話題か

2026-05-18（Google I/O 2026）に出た Flutter 3.44.0 / Dart 3.12 は、機能追加より「土台の作り替え」が主役のリリースだ。これまで SDK 本体に同梱されていた UI レイヤと iOS/macOS のビルド基盤が、より独立した形へ動いた。3ヶ月ごとの SDK サイクルに縛られず UI 改善を受け取れるようになる、という開発リズムの変化が肝になる。

何が新しいか

最大の変化は Material / Cupertino の本体リポジトリからの分離。material_ui / cupertino_ui として pub.dev で配布され、Material 3 の新機能や Cupertino の修正を SDK リリースを待たずに随時取り込める。Cupertino には iOS ネイティブ感のある CupertinoMenuAnchor が追加。ビルド面では iOS/macOS の既定依存管理が SwiftPM（Swift Package Manager）に昇格し CocoaPods 終焉へ向かう。副産物として Apple Silicon Mac で Rosetta 不要になった。Widget Preview は Dart Analysis Server ベースへ刷新され IDE 利用時メモリを最大 50%削減。Dart 3.12 は Agentic Hot Reload と Genkit で AI/エージェント対応を強化し、private named parameters が正式、primary constructors が実験的に入った。

状態管理とパフォーマンスの現在地

新規プロジェクトは Riverpod 3.3.1 + コード生成が推奨デフォルト。コンパイル時安全・context-free read・first-class async に加え、3.0系で Mutations API・指数バックオフの自動リトライ・stale-while-revalidate のオフライン永続化が本番投入された。監査証跡が要る規制業界は Bloc 9.x、GetX は12ヶ月超のプロジェクトでは非推奨という棲み分け。描画は Impeller が iOS/モダン Android で既定（API 29+、Flutter 3.27以降）。シェーダー事前コンパイルで初回フレームからジャンクが無く、ラスタライズ約50%高速化・安定 120fps を狙える。

自分が試すなら

  • iOS プロジェクトを SwiftPM 移行し、CocoaPods 依存と Rosetta 起動を外してビルドを軽くする。
  • material_ui/cupertino_ui を pub 依存に切り出し、Material 3 更新を SDK と切り離して追えるようにする。
  • パフォーマンス検証は必ず profile モードで。DevTools の Performance タブでラスタライズ <8.3ms（120fps予算）を確認する。

注意点

UI パッケージ分離は柔軟さと引き換えに依存バージョンの管理対象が増える。pub の更新と SDK の更新が別軸で動くので、CI でのバージョン固定とアップグレード手順を決めておきたい。SwiftPM 移行も既存の CocoaPods 前提のプラグインが残っていると詰まりやすい。

ひとことまとめ

3.44 は「速くなった」より「分離して進化が速くなる土台ができた」リリース。Riverpod 3 と Impeller が安定路線に乗った今、新規はこの構成を素直に採るのが堅い。
