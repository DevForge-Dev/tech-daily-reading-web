フィードバックを「設計」する — Next.js App Routerのマイクロインタラクション実装

hover/press/focusの3状態、スケルトン、リップル、トースト。多くはCSSだけで60fpsで完結し、Server Componentのまま使える。そこにReact 19のuseFormStatus・useActionState・useOptimisticを噛み合わせる。reduced-motion配慮まで含めた「反応するUI」の作法を実装目線で。

📅 2026年6月18日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

マイクロインタラクションとは、ホバー・押下・ローディング・成功/失敗といった「操作への小さな視覚フィードバック」のこと。1つ数百msの演出だが、UIが反応している感覚＝知覚パフォーマンスと信頼感を大きく左右する。重要なのは、大半がCSSだけ（transition / keyframes）で GPU 合成・60fps で完結し、'use client' 無しの Server Component のまま使える点。そこに Next.js App Router 側の「フィードバック専用の仕組み」を組み合わせるのが今どきの作法だ。

何が新しいか（CSSの要点）

基本は3状態。ホバーは scale(1.02〜1.05) 程度に抑え、押下は :active で scale(.96)＋影を弱めて「カチッ」と沈ませ（押下時だけ 80〜120ms と短く）、フォーカスは必ず :focus-visible でリングを出す。ローディングはスピナーよりスケルトンが知覚速度で勝り、実コンテンツと同じ寸法にして CLS を防ぐ。シマーは @keyframes 1個で background-position を流すだけ。リップルは transform: scale() ＋ opacity のみで実装し、合成だけで完結させる。

React 19 と噛み合わせる「送信中」演出

状態に応じて切り替わる部分だけを Client Component にする。useFormStatus（react-dom）は <form> の子で pending を取り、ボタンを spinner＋disabled に。フォーム本体は Server Component のままでよい。useActionState（React 19）は Server Action の結果・エラー・pending を1つの state にまとめ、バリデーション表示と送信中演出を一括で扱う。useOptimistic はサーバ応答を待たず即 UI 反映し、失敗時に自動ロールバック（「いいね」等で体感速度が激変）。ルート遷移時は loading.tsx のスケルトンが静的シェルとして先行配信される。

落とし穴と reduced-motion

アニメ対象は transform/opacity に寄せる。useFormStatus はフォームの子でしか効かない（同階層・親では pending が取れない）。スケルトンと実体の寸法ズレは CLS を生む。過剰演出は逆効果で、1操作1フィードバック・概ね 150〜500ms に収める。prefers-reduced-motion: reduce では動きは消すが意味は残す——シマーは静的グレー、押下の沈み込みは色変化で代替、送信中は spinner ではなく「送信中…」テキスト＋ aria-busy で十分伝わる。

ひとことまとめ

「反応するUI」はライブラリではなく、CSSの3状態＋Reactの状態フックの組み合わせで作れる。Server Component を保ちつつ、まずは送信ボタンの useFormStatus と一覧の loading.tsx スケルトンから始めるのが入りやすい。
