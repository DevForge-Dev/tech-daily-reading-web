nuqs 2.5 — URLを「第3の状態置き場」にする型安全searchParams管理

フィルタ・検索語・ページ番号のような「共有できるべきUI状態」の正しい置き場はZustandでもTanStack QueryでもなくURL。nuqsはそれをuseState互換のAPIで書けるようにする5.7kBのライブラリだ。

📅 2026年7月13日 ⏱ 約3分 🔗 出典 4 件

Zustand（クライアント状態）とTanStack Query（サーバー状態）を押さえると、残る問題は「フィルタや検索語やタブ選択をどこに置くか」だ。これらはリロードしても共有リンクでも復元されてほしい状態で、正しい置き場はURLクエリ文字列になる。ただし素のuseSearchParams + router.pushは文字列変換とシリアライズのboilerplate地獄になりがち。nuqs（5.7kB gzip）はそこを「useStateのURL版」という一点突破で解決する。

APIはuseQueryState('q', parseAsString.withDefault(''))が[value, setValue]を返すuseState互換。integer/boolean/isoDateTime/enum/jsonなどのパーサーを宣言的にチェーンする。v2ではルートレイアウトをNuqsAdapterで包むのが必須（忘れるとランタイムエラー）。鍵になるオプションがshallowで、既定のtrueならURL更新はブラウザ内だけ、falseにするとNext.jsに通知されてsearchParamsを受けるRSCページが再レンダー＝サーバー側フィルタリングと連動する。

const [{ q, sort }, setFilters] = useQueryStates(productSearchParams, { shallow: false })
// 検索入力ではURL書き込みをdebounce（stateは即時反映）
setFilters({ q: value, page: 1 }, { limitUrlUpdates: debounce(300) })

サーバー側はnuqs/serverからcreateLoader(parsers)でページやRoute Handlerを型安全にパースでき、createSearchParamsCacheを使えば深くネストしたServer Componentからも参照可能（RSC版Contextに相当）。loadSearchParamsをawaitせずPromiseのままSuspense内の子へ渡せばPPR/ストリーミングとも噛み合う。2.5.0ではlimitUrlUpdates: debounce(ms)（旧throttleMsを置換）、パーサー定義からバリデーションスキーマを導出するStandard Schema生成、キー単位サブスクリプションで再レンダーを減らすkey isolationが入った。

自分が試すなら、一覧ページのフィルタ群をパーサー定義1ファイルに集約し、RSCページとClient島で同じ定義を共有する構成。落とし穴は4つ——パーサーはバリデーションではない（「正の整数のみ」等はZodで後段検証）、useQueryStateを使うコンポーネントは'use client'必須、shallow: falseは毎回サーバーリクエストなので検索入力では必ずdebounce併用、そしてclearOnDefaultが既定trueなのでdefault値と一致するとURLからキーが消える（意図的な仕様）。

ひとことまとめ: 「この状態は共有できるべきか？」と自問してYesならnuqs。Zustand・TanStack Queryと合わせ、状態の置き場所の三角形がこれで完成する。
