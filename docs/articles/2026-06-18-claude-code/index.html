Dynamic Workflows 正式発表 — 1セッションで数十〜数百のサブエージェントを自動オーケストレーション

Claudeが依頼からJavaScriptのオーケストレーションスクリプトを自動生成し、単一セッション内で数十〜数百の並列サブエージェントを起動・検証する。「単一エージェントには大きすぎる」広範なバグ調査・大規模マイグレーション・監査を自動化する新機能を、v2.1.179の修正や6/15の課金分離とあわせて読み解く。

📅 2026年6月18日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

サブエージェントは便利だが、これまでは人間（やメイン会話）がタスクを分解し、委譲し、結果を統合する必要があった。対象がモノレポ全体・数十ファイル横断のように大きくなると、この手作業のオーケストレーションが現実的でなくなる。Anthropic が Claude Code に導入した Dynamic Workflows（動的ワークフロー）は、この「分解→並列実行→検証→統合」の段取りそのものを Claude に任せる仕組みだ。スケジュールタスクや CI で大型作業を回す人ほど効いてくる。

何が新しいか

依頼内容から Claude が JavaScript のオーケストレーションスクリプトを自動生成し、ランタイムがそれをバックグラウンドで実行する。1セッション内で数十〜数百の並列サブエージェントを起動し、各サブタスクの結果を検証してから最終回答を返す。ランタイムが裏で走るためメインセッションは応答性を保つ。提供は Max・Team・対象 Enterprise でリサーチプレビュー、API / Bedrock / Vertex / Foundry でも利用できる。広範なバグ調査・大規模リファクタ・セキュリティ監査など「単一エージェントには大きすぎる」タスクが射程に入る。

仕組みのポイント

肝は「スクリプトとして書き出される」こと。分解・並列・検証のパイプラインが明示的なコードになるため、確率的なモデルの解釈に丸投げするのではなく決定論的な制御を効かせやすい。従来の原則「1エージェント1ジョブ」を、多数束ねて動かすイメージに近い。冗長な中間出力は各サブエージェントの文脈に隔離され、メインには結論だけが返るので文脈も汚れにくい。

自分が試すなら

  • モノレポ横断の依存アップグレード調査や、コードベース全体のバグ探索を1依頼で投げてみる。
  • セキュリティ監査や大規模マイグレーションの「下調べ」を並列サブエージェントに分担させ、結論だけ受け取る。
  • /usage のカテゴリ別内訳で、並列実行のトークン消費を必ず監視する。

注意点

数百サブエージェントはコストが跳ねる。6/15 の課金分離で自動化（Programmatic）は対話枠と別管理、Agent SDK は別クレジット消費になった。報道では Microsoft が直接ライセンスを縮小し Copilot CLI へ誘導、Uber は年間予算を4カ月で使い切ったとされる。規模と費用の見積もりは先に。なお同時期の v2.1.179 は、中断接続時に生エラーではなく部分応答を保持し、「running tool」でのスピナー固着や WSL2 スクロールを修正している。

ひとことまとめ

Dynamic Workflows は Claude Code を「1タスクを手伝う道具」から「大型タスクを自走でこなすランタイム」へ押し上げる。鍵は決定論的なスクリプト制御とコスト管理。まずは小さめの調査タスクで挙動と消費感をつかむのが堅い。
