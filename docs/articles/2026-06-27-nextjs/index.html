JSなしのネイティブCSSカルーセル — ::scroll-button() / ::scroll-marker で Server Component のまま送りボタンとドットを作る（Chrome/Edge 135）

カルーセルはこれまで embla や swiper などクライアントJS必須の代表格で、App Router では 'use client' 化とhydrationのコストを払ってきた。CSS Overflow Module Level 5 のプリミティブが Chrome / Edge 135（2025年3月）で出荷され、JavaScriptなしで送りボタンとドットページャを付けられるようになった。Server Component のまま動き、端での自動 disabled やキーボード巡回まで標準装備。ただし対応はChromium系のみで、プログレッシブエンハンスメント前提で使う。

なぜ今これが話題か

カルーセルはこれまで embla や swiper などクライアントJS必須の代表格で、App Router では 'use client' 化と hydration のコストを払ってきた。CSS Overflow Module Level 5 が定義した一連のプリミティブが Chrome / Edge 135（2025年3月）で出荷され、JavaScriptなしで送りボタンとドットページャを付けられるようになった。Server Component のまま動くので、クライアントJSもhydrationも不要になり、バンドル削減とLCP/INPに効く。

何が新しいか

カルーセルの正体は「スクロール領域＋最大2つのアフォーダンス」。土台は普通のオーバーフロー領域に scroll-snap-type: x mandatory を付けるだけ。そこへ擬似要素を足す。::scroll-button(left/right) はブラウザ生成の本物の <button> で、押すとスクロール領域の85%送り（snapがあれば実質1枚送り）、端で自動的に :disabled になる。第2引数で代替テキストも渡せる（content: "➡" / "次へ"）。scroll-marker-group でドット/サムネの入れ物を生成し、各アイテムの ::scroll-marker がその項目へのアンカー <a> を作る。現在地は :target-current で装飾でき、キーボード巡回やスクリーンリーダーの tablist 相当の読み上げ、端での自動無効化までブラウザが面倒を見る。

App Router での組み込みとフォールバック

純CSSなので 'use client' 不要、RSCで map して生成できる。最大の注意は対応範囲で、2026年6月時点でも Chromium 系のみ（Firefox/Safari は実装中）。未対応ブラウザでは擬似要素が無視されるが、素のスクロール領域としては成立する。だからプログレッシブエンハンスメント前提で設計し、どうしても全ブラウザでボタン/ドットが要る場合だけ CSS.supports('selector(::scroll-marker)') で判定して未対応時に小さなクライアントコンポーネントで補う（＝対応ブラウザはJSゼロ）。スムーススクロールは prefers-reduced-motion: no-preference のときだけ有効化し、機能（送り・ジャンプ）は常に残す。

自分が試すなら

  • RSC のまま <ul class="carousel"> を map で出し、::scroll-button() と ::scroll-marker をCSSだけで足す最小カルーセルを組む。
  • Tailwind v4 を使う場合は、擬似要素は @layer で素のCSSを書き、配色は @theme トークンを var() 参照で統一する。
  • Firefox/Safari で素のスクロール領域として成立するかを必ず確認し、UI前提にしない。

注意点

対応は Chromium 135+ のみ。scroll-behavior: smooth × snap × 85%送りは相性で崩れることがあるので grid-auto-columns と gap で調整する。v1では自前要素の差し込み・循環スクロールは不可（将来対応）。「カルーセル」と呼ぶが本質はスクロール領域のナビ強化で、タブUI・スライド・ギャラリーにも同じ仕組みが使える（過剰にカルーセル化しない）。

ひとことまとめ

ネイティブCSSカルーセルは「JSを足さずにアクセシブルな送り＆ページャを得る」手段。Server Component と相性が良くバンドルとCWVに効くが、当面はChromium限定なので、未対応でも成立する設計を土台に進行的強化として載せるのが正解だ。
