Zustand を Next.js App Router で安全に — グローバルストアの罠と Store Factory + Context パターン

Zustand は create した hook をどこからでも呼べる手軽さが売りだが、App Router の SSR ではこのグローバルストアがそのまま落とし穴になる。サーバは1プロセスで複数リクエストを捌くため、モジュールスコープのストアを共有するとユーザーAの状態がユーザーBに漏れる。公式の解は「リクエストごとにストアを生成する Store Factory ＋ React Context で配布」。RSCはストアに触らず、クライアント状態は Client Component に閉じる、という線引きが軸になる。

なぜ今これが話題か

Zustand は軽量・ボイラープレート最小で人気の状態管理ライブラリ。基本は「ストア＝モジュールのグローバル変数」で、create した hook を Context 不要でどこからでも呼べる手軽さが売りだ。ところが Next.js（特に App Router の SSR）では、このグローバルストアがそのまま落とし穴になる。状態管理カテゴリの基礎として最初に押さえたい論点だ。

何が問題か

サーバは1プロセスで複数リクエストを同時に捌く。モジュールトップで create(...) したストアをアプリ全体で共有すると、ユーザーAの状態がユーザーBに漏れる（クロスリクエスト汚染＝実質セキュリティ事故）。加えて SSR とクライアントで初期値が食い違うと hydration error（Text content does not match...）になる。原因の多くは localStorage/persist・Date.now()・window など「クライアントだけが持つ値」を初回レンダリングで描いてしまうことだ。

公式パターン：Store Factory + Context

解は2段構え。①ストア本体を zustand/vanilla の createStore を返すファクトリ関数として書く（'use client' 不要の純ロジック）。②'use client' の Provider で useState(() => createStore()) として「マウント時に1回だけ」生成し、Context で配る。useState の初期化関数にすることで、再レンダリングでも作り直さず1インスタンスに安定する。RSC はストアに触らない——Server Component は hooks も Context も使えず、読み書きは Client Component に閉じる。

// store（純ロジック・'use client'不要）
export const createCounterStore = (init = { count: 0 }) =>
  createStore<CounterStore>()((set) => ({
    ...init,
    inc: () => set((s) => ({ count: s.count + 1 })),
  }))

// provider（'use client'）— マウント時1回だけ生成
const [store] = useState(() => createCounterStore())

自分が試すなら / 注意点

  • サーバ由来データで初期化したいときは RSC で取得→props で Provider に渡し、ファクトリの initState へ流す（初回HTMLとクライアント初期化が一致し hydration mismatch を防ぐ）。
  • 複数フィールドを1セレクタで返すと毎回新オブジェクトで再描画が増える。useShallow か、フィールドごとにセレクタを分ける。
persist（localStorage）は初回サーバレンダリングに存在せず hydration mismatch の常連。skipHydration/onRehydrateStorage でマウント後に反映する。Server Actions の楽観的更新は useOptimistic に任せ、Zustand はモーダル開閉・フィルタ等の純粋なUI状態に絞ると責務が綺麗だ。

ひとことまとめ

「グローバル create を共有しない／ファクトリで生成／Providerは 'use client'／RSCは触らない」が要点。まずは Counter 1個を Store Factory で組むと型が掴める。
