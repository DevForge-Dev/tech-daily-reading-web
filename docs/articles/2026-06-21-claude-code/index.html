Claude Code

Claude Platform に Workload Identity Federation が正式提供 — 静的APIキーを短命OIDCトークンへ、Claude Code/CI が鍵レスに

エージェントやCIにコードを任せる時代に最後まで残る弱点が静的APIキーだ。6/21、Anthropic は Workload Identity Federation（WIF）を Claude Platform で一般提供（GA）にした。長寿命の sk-ant-… を、数分で失効する短命OIDCトークンに置き換える仕組みで、Claude Code 経由のアクセスにも適用される。claude-code-action なら ANTHROPIC_API_KEY シークレットすら不要になる。

📅 2026年6月21日 ⏱ 約3分 🔗 出典 3 件

なぜ今これが話題か

エージェントやCIにコードを任せる時代に、最後まで残る弱点が静的APIキーだ。sk-ant-… の長寿命キーは、漏洩すれば即悪用され、定期ローテーションの運用負担も重い。これを根本から変えるのが Workload Identity Federation（WIF）で、本日 Claude Platform で一般提供（GA）になった。

何が新しいか

WIF は長寿命の静的キーの代わりに、OIDC準拠のIdPが発行する数分で失効する短命・スコープ付きトークンでワークロードを認証する。対応発行者は AWS IAM・Google Cloud・GitHub Actions・Kubernetes・Entra ID・Okta など標準OIDC。全Claude APIエンドポイントを網羅し、Claude Code 経由のアクセスにも適用される。鍵の作成・保管・ローテーション・漏洩リスクをまとめて排除できる。

GitHub Actions では鍵レス化

公式の claude-code-action も WIF に対応した。ワークフローの GitHub Actions OIDC トークンを短命の Anthropic アクセストークンに交換することで、ANTHROPIC_API_KEY シークレットの作成・保存・ローテーションが不要になる。PRレビューや自動修正をCIに組み込むチームほど、鍵管理の負担と漏洩面が小さくなる。

仕組みのポイント

核心は「信頼の確立をIdPに委ねる」こと。Anthropic 側は事前に信頼するIdPと条件（リポジトリ・ブランチ等のclaim）を登録し、実行時にワークロードが提示するOIDCトークンを検証して短命トークンを発行する。鍵という"持ち物"を、"その場で証明する身元"に置き換えるイメージだ。Hookや設定と違い、認証基盤側のガードなので追加コードなしで効く。

自分が試すなら

  • CIの ANTHROPIC_API_KEY シークレットを廃し、claude-code-action のOIDC交換へ移行する。
  • IdP側で信頼条件（対象リポジトリ/ブランチ）を最小限に絞る。
  • サーバ常駐やスケジュール実行の自動化も、可能ならWIF発行の短命トークンへ寄せる。

注意点

WIFは万能ではない。IdP側の信頼設定やトークン交換の運用を誤れば別の穴になり、非OIDC環境では恩恵を受けにくい（セキュリティ企業Aembitも利点と残課題の両面を指摘）。最新版 Claude Code は 6/21 時点でも v2.1.183 据え置きで、本件はプラットフォーム側の変更である点も押さえたい。

ひとことまとめ

「鍵を持たない」のが最も安全な鍵管理。WIFのGAで、Claude Code もCIも短命トークンで回せるようになった。まずはCIの静的キー廃止から着手するのが効果的だ。
