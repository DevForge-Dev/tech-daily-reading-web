Flutterが単一エンジンへ収束 — Impeller をデスクトップへ展開し6面を統一、Web は WASM 既定化、GenUI SDK＋A2UI で UI を動的生成

2026後半ロードマップ「Finishing the Job」は、Android の Impeller 移行完了に加え、Impeller をデスクトップ(macOS/Windows/Linux)へ展開して6プラットフォームを単一の描画エンジンに統一する方向。Web は WASM を既定コンパイルターゲットへ前進し、GenUI SDK＋A2UI プロトコルで AI が動的に UI を生成する。安定版は Flutter 3.44.0 / Dart 3.12。

なぜ今これが話題か

マルチプラットフォーム対応の代償は「環境ごとの描画差」だった。2026後半ロードマップ 「Finishing the Job」 は、その分断を終わらせにいく。6プラットフォームを単一の描画エンジンに統一する方向が打ち出された。安定版は Flutter 3.44.0 / Dart 3.12（Google I/O 2026, 2026-05-18）。

何が新しいか

第一に、Android の Impeller 移行完了（Android 10+ で Skia 撤去）に加え、Impeller をデスクトップ（macOS/Windows/Linux）へ展開し、モバイル・デスクトップ・Web を含む全プラットフォームを1つの描画エンジンへ収束させる。第二に、Flutter Web は WASM を既定コンパイルターゲットへ前進（現状は --wasm フラグ必須、3.35+ で Wasm dry-run により切替を準備中）。第三に、GenUI SDK が pub.dev で alpha 提供（Flutter 3.38 投入）され、A2UI プロトコルで AI モデルが Flutter の Widget カタログを使い、テキストでなく動的に UI を生成する。

仕組みのポイント

Impeller はシェーダーをビルド時に AOT 事前コンパイルし、従来の弱点だったシェーダーコンパイルジャンクを解消する。これをデスクトップにも広げることで、6面が同じ描画パイプラインに乗り挙動の予測可能性が上がる。GenUI/A2UI は、モデルが自由テキストを返すのではなく用意された Widget 語彙の範囲で UI を構築する点が肝で、エージェントが画面そのものを生成する「エージェント志向 UI」に現実味を与える。

自分が試すなら

  • 既存デスクトップアプリを Impeller で動かし、描画差や性能を profile モードで比較する。
  • Web 配信のアプリで --wasm ビルドを試し、体感・サイズ・性能を計測する。
  • GenUI SDK の alpha を小さな動的フォーム生成で触り、A2UI の語彙設計を試す。

注意点

デスクトップ Impeller や Web WASM 既定化は移行途上で、現時点では --wasm フラグや --no-enable-impeller での切り分けが必要な場面が残る。言語面は Dart 3.13 で primary constructors が正式追加見込みなど進化が速いので、最新の digests/ の日付を確認してから採用判断したい。GenUI/A2UI は alpha のため本番採用は時期尚早だ。

ひとことまとめ

「6面を1エンジンに」「Web は WASM」「UI は AI が生成」という3方向の収束が今年のテーマ。まずはデスクトップ Impeller と Web WASM の計測から触れておくと流れに乗れる。
