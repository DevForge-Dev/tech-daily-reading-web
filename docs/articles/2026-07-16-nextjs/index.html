Auth.js v5 を App Router で — 現行セットアップと2026年の地殻変動（Better Auth移行・middleware→proxy改名）

2025/9/22にAuth.jsはBetter Authチーム管理下へ入り実質メンテナンスモードに。新規はBetter Auth推奨、例外はDB無しステートレスセッション。Next.js 16でmiddleware.ts→proxy.tsに改名。単一auth.tsが{handlers,signIn,signOut,auth}を返し、auth()をRSC/Handler/Action/proxyで共用。proxyは境界でない（CVE-2025-29927）ためDALで再検証。

📅 2026年7月16日 ⏱ 約3分 🔗 出典 5 件

Auth.js（旧 NextAuth）は App Router 時代の定番認証ライブラリだが、2026年の今これを扱う理由は「使い方」以上に立ち位置の激変を正しく知るためだ。

何が新しいか（地殻変動）

2025年9月22日、Auth.js は Better Auth チームの管理下に入った。以後 next-auth/Auth.js は実質メンテナンスモード＝セキュリティパッチと緊急対応のみで、新機能は基本入らない。Better Auth 公式は「新規は Better Auth を強く推奨」と明言するが、例外としてDB無しのステートレスセッションは現状 Auth.js の強み。さらに Better Auth 自体がVercel に参画し、認証は Next.js/Vercel 側に一層寄っていく。加えて Next.js 16 では middleware.ts が proxy.ts に改名され、ガード用エクスポートも export { auth as proxy } になった。

仕組み（v5 の型）

設定はプロジェクト直下の単一 auth.ts に集約し、NextAuth(config) が { handlers, signIn, signOut, auth } を返す。中でも auth() が万能で、Server Components / Route Handlers / Server Actions / proxy のすべてで同じ関数からセッションを読める。既定のセッション戦略はJWT（暗号化Cookie・DBクエリ不要）だ。

// ./auth.ts
export const { handlers, signIn, signOut, auth } = NextAuth({
  providers: [GitHub],
  session: { strategy: "jwt" }, // DB無しステートレス＝Auth.jsの強み
})

// ./proxy.ts （Next.js 16 / 旧 middleware.ts）
export { auth as proxy } from "@/auth"

落とし穴

最大の注意は proxy（旧middleware）はセキュリティ境界ではないこと。CVE-2025-29927（ヘッダ偽装で middleware 認可をバイパス）がこれを決定づけた。認可は必ず DAL（Data Access Layer）で auth()/verifySession() を使って再検証する。Credentials プロバイダはJWT戦略必須、ロールは jwt/session コールバックで明示的に詰め替える（自動では入らない）。クライアントで useSession() を使うなら <SessionProvider> でラップが要る（RSC では auth() 直呼びで Provider 不要）。

自分が試すなら

既存 Auth.js アプリはそのまま使い続けてよい（壊れない）。新規は Better Auth を第一候補に検討し、Auth.js を選ぶなら「DBを持たず JWT だけで回したい」等の明確な理由を持つ。まずは npx auth secret で AUTH_SECRET を生成し、auth.ts ＋ Route Handler ＋ proxy.ts の最小構成を組んで、DAL 再検証をテンプレ化しておくのが安全だ。

ひとことまとめ

「Auth.js は延命、新規は Better Auth」。そして認可はデータの隣で——proxy に守ってもらうな。
