shadcn/ui はもう「コンポーネント集」ではない——レジストリ配布プロトコルへ（CLI v4）

shadcn/ui は npm 公開ではなく、CLI がコード（.tsx/CSS変数/設定）を自分のリポジトリに書き込む "open code" 方式。2026年は @registry/name 名前空間・private registry・init --preset・registry:base/font・shadcn/skills まで備えた分散配布プロトコルに拡張。App Router では表示系は RSC のまま、イベント持ちは 'use client' 済みの Client 島として動く。

📅 2026年7月20日 ⏱ 約3分 🔗 出典 3 件

この日のNext.jsテーマはshadcn/uiの実像だ。もはや「npmに公開されるUIライブラリ」ではなく、CLIがコード（.tsx／CSS変数／設定）を自分のリポジトリに書き込む "open code" 方式で、生成後は100%自分の所有物として直接編集できる。2026年、これは単なるコピー機を超えてコンポーネント・テーマ・デザインシステム丸ごと・AIエージェント向け指示までを配る分散配布プロトコルへ拡張された。App Routerはshadcn公式が第一級に扱うため、押さえる価値が高い。

3つの節目

CLI 3.0（2025-08）で@registry/name形式の名前空間・private registry・view/search/list・MCPサーバ・解決エンジン全面刷新。Unified Radix／Tailwind v4（2026-02）で個別@radix-ui/react-*が単一radix-uiへ集約され、全プリミティブにdata-slotが付いた。CLI v4（2026-03）でinit --preset（デザインシステム設定を短い文字列1本に凝縮）・--dry-run/--diff・registry:base/registry:font・shadcn/skillsが加わった。

仕組みのポイント

名前空間はcomponents.jsonのregistriesにエイリアスを宣言し、{name}が導入時にリソース名へ置換される。private registryはheadersに${REGISTRY_TOKEN}を書けば実行時展開され、ログには出さない。registryDependenciesに@shadcn/cardや@acme/data-tableを混在でき、CLIが各ソースから再帰解決＋トポロジカルソート＋同一パスは後勝ちでマージする。レジストリのリソースはデータ（JSON）でありスクリプトではないため、任意コード実行はない。

App Routerでの組み込み

pnpm dlx shadcn@latest initでcomponents.jsonを生成し、add button card dialogでコードがcomponents/ui/*.tsxに書き込まれる。表示系（Card/Badge/Table）はRSCのまま使え、Dialog/Dropdown等イベント持ちは先頭に 'use client' が既に入ったClient島として動く。更新はnpmではなくadd --diffで手元改変とマージ追従する運用だ。

落とし穴

後勝ちマージの順序、信頼するレジストリだけを明示追加する点（自動発見なし・ENVトークンはコミットしない）、--base radixと--base base（Base UI）で実装/APIが異なる点に注意。生成コードは直接編集する前提なので、上流更新は手動マージになる。

ひとことまとめ

「shadcn/uiはUIライブラリではなく配布プロトコル」。private registry＋registry:baseで社内標準を1行の名前空間として配れるのが2026年の勘所。
