App Router の認可設計 — Data Access Layer × proxy.ts × Server Actions で「データに最も近い場所」を守る

App Router の認証・認可は「どこでチェックするか」を間違えると、ページは守れても Server Actions や子コンポーネントから素通りされる。公式が一貫して推すのは多層防御＋データに最も近い場所での認可で、その中心が Data Access Layer（DAL）に verifySession() を集約するパターンだ。Next.js 16 で middleware.ts は proxy.ts に改名され、CVE-2025-29927（CVSS 9.1）の middleware バイパス事件が「middleware に認可を依存するな」という設計を決定づけた。

なぜ今これが話題か

App Router では認可をどこに置くかで安全性が決まる。Layout やページの入口だけで return null する SPA 的な発想だと、ネストした route segment・Route Handler・Server Actions が止まらず素通りされる。公式は一貫して多層防御（defense-in-depth）＋データに最も近い場所での認可を推し、その本丸が Data Access Layer（DAL）だ。加えて Next.js 16 で middleware.ts が proxy.ts に改名され（公式認証ガイド v16.2.9 / 2026-06-23 更新）、CVE-2025-29927（CVSS 9.1, 2025-03公開）の middleware 認可バイパス事件が設計指針を固めた。

何が新しいか

認可は2種類に分けて考える。楽観的（Optimistic）は Cookie 内のセッションだけで判定し、UIの出し分けやリダイレクトなど軽い操作向けで proxy.ts が担う（DBには触らない——proxy はprefetch含め全ルートで走るためDBアクセスは性能事故になる）。厳密（Secure）は DB のセッション実体と突き合わせ、機微データ・重要操作向けで DAL が担う。proxy.ts は「最初の門番」であって最後の砦ではない。CVE-2025-29927 は x-middleware-subrequest ヘッダ偽装で middleware/proxy の認可を完全バイパスできた事件で、next start + output: 'standalone' のセルフホスト構成が影響対象（Vercel/Netlify ホストは非影響、15.2.3+ / 14.2.25+ 等で修正済）。だからセキュリティチェックはデータソースに最も近い場所へ寄せる。

DAL に認可を集約する

核は verifySession() を専用ファイルに置き、React の cache() でメモ化して1レンダーパス内の重複検証・重複クエリを防ぐこと。ファイル先頭に import 'server-only' を置けば、認証・DBロジックがクライアントへ混入するのを物理的に遮断できる。getUser() 等のデータ取得関数は必ず先に verifySession() を呼ぶことで「呼べばどこでも認可が走る」状態を担保する。返却はDTO で必要な列だけに絞り、password や電話番号の漏洩を防ぐ。

// app/lib/dal.ts
import 'server-only'
import { cache } from 'react'
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { decrypt } from '@/app/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)
  if (!session?.userId) redirect('/login') // 未認証は弾く
  return { isAuth: true, userId: session.userId }
})

自分が試すなら

  • 既存プロジェクトの認可ロジックを app/lib/dal.ts に集約し、各データ取得関数の先頭で verifySession() を呼ぶ形に揃える。
  • 各 Server Action / Route Handler の先頭にも verifySession() + ロール確認を入れる（Route Handler は未認証=401、権限不足=403 の二段）。
  • セルフホスト構成なら Next.js のバージョンを確認し、CVE-2025-29927 修正済み（15.2.3+ 等）かを点検する。

注意点

Layout はPartial Renderingでナビゲーション時に再レンダーされない＝ルート遷移ごとにセッションが再検証されないため、Layout で認可を完結させない。'use server' はサーバへ移すモジュール境界の指示にすぎず、認証・入力検証・レート制限を一切付与しない（markした関数は誰でも叩けるPOSTエンドポイント）。トークンは localStorage に置かず httpOnly Cookie（Secure / SameSite / Path 付き）を使い、セッションJWTのペイロードは userId・role など最小限に絞る。

ひとことまとめ

App Router の認可は「入口で守る」発想を捨て、データに最も近い DAL で verifySession() を必ず通すのが正解。proxy.ts は楽観的な門番、Server Actions / Route Handler は公開APIとして各自で認可。CVE-2025-29927 が示した通り、middleware/proxy を唯一の防御にしてはいけない。
