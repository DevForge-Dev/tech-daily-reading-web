アニメーションを「予算内」で設計する — タイミング階層・Staggered・Impellerとフレーム予算

micro 50〜200ms=implicit、component 200〜400ms=explicit、Hero 300〜600ms。複数アニメは単一AnimationControllerにIntervalで束ねるStaggeredが定石。Impellerのシェーダー事前コンパイルと、60/90/120fpsのフレーム予算を軸に「カクつかない動き」を組む。

📅 2026年6月18日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

120Hz 端末が主流になり、1フレームの予算は8ms 台まで縮んだ。動きの良し悪しは派手さではなく、タイミング設計と再ビルド削減で決まる。現行の安定版は Flutter 3.44 / Dart 3.12。実装の勘どころは「用途に応じた継続時間」と「フレーム予算を超えない描画」の2点に集約される。

何が新しいか（タイミング階層とStaggered）

まず用途で継続時間を決めるのが定石だ。micro 50〜200ms は implicit（AnimatedContainer 等）、component 200〜400ms は explicit（AnimationController）、Hero 300〜600ms は flightShuttleBuilder で独自フライトを描く。複数要素を時間差で動かす Staggered（時間差）アニメは、個別にコントローラを持たせず単一の AnimationController にまとめ、Interval で各要素の開始/終了を全体 duration の中に配置する。リストやグリッドの演出は flutter_staggered_animations、定型コード削減は flutter_hooks（HookWidget）が手軽だ。

Impellerとフレーム予算

描画エンジン Impeller は、シェーダーをビルド時に AOT 事前コンパイルするため、複雑シーンでラスタライズ約50%短縮・初回フレームからジャンクが出にくい。フレーム予算は 60Hz=ビルド16ms・描画16ms、120fps 端末では合計 8ms 未満を狙う。Rive を使うなら 0.12+ の events で名前付きイベントを受け取り、.riv は initState/FutureBuilder で事前ロードして build() 内ではロードしない。

自分が試すなら

  • 検証は必ず profile モードで行う（debug VM は大幅に遅く、数値が当てにならない）。
  • DevTools の widget rebuild tracking で各フレームの時間消費を特定し、const 付与で再ビルドをスキップ、Consumer/Selector で局所再描画にする。
  • 画像は cached_network_image＋width/height 明示でレイアウト揺れと再デコードを防ぐ。

注意点

最頻出の性能問題は不要な再ビルド。大きな build() は分割し、アニメ中に親ごと再ビルドしない設計にする。Impeller で描画差が出たら --no-enable-impeller で Skia と比較して切り分ける。.riv を build() 内でロードするとフレーム落ちの原因になるので必ず事前ロードする。

ひとことまとめ

滑らかなアニメは「用途で継続時間を決め、単一コントローラで束ね、フレーム予算内に描く」の3点で作れる。まずは profile モードでの計測と const 化から、予算内に収める癖をつけるのが近道だ。
