Flutter、Android の Skia を撤去 — Pure Impeller(Vulkan) が唯一の描画経路に、シェーダーは事前コンパイル

複数年がかりの描画パイプライン移行がついに決着。Android 10 以上で Skia バックエンドが撤去され、Pure Impeller(Vulkan) が既定かつ唯一の選択肢になった。全シェーダーはビルド時に SPIR-V へ事前コンパイルされ、起動時のシェーダーコンパイル由来のスタッターが消滅。CustomPaint も Metal/Vulkan で大幅に速くなる。

なぜ今これが話題か

Flutter で長年の悩みだったのが シェーダーコンパイル由来のジャンクだ。Skia は初回に出会った描画を実行時にコンパイルするため、最初のアニメや遷移でカクッと一瞬詰まることがあった。これを根本から断つのが描画エンジン Impeller で、その移行が 2026 についに決着した。

何が新しいか

Android 10 以上で Skia バックエンドが撤去され、Pure Impeller（Vulkan）が既定かつ唯一の描画経路になった。複数年計画の総決算だ。Impeller は描画プリミティブが有限である前提で、エンジンが使う全シェーダーをビルド時に SPIR-V として事前コンパイルする。起動時点でシェーダーが揃っているのでバックグラウンドのシェーダーコンパイルが消滅し、予期せぬスタッターのない 60/90/120fps を狙える。安定版は Flutter 3.44.0（2026-05-18）/ Dart 3.12.1（2026-05-26）。

仕組みのポイント

Skia は OpenGL 時代の汎用 2D エンジンで、何でも描ける代わりに実行時コンパイルが避けられなかった。Impeller は「Flutter が実際に使う描画は有限」と割り切り、必要なシェーダーをAOT（事前）で用意する。さらに Metal(iOS)/Vulkan(Android) という低レベルで効率的な GPU 経路を直接叩く。結果、複雑シーンで平均ラスタライズ時間が約50%短縮、Skia が遷移時に 16ms 超へスパイクする場面でも Impeller は120Hz に必要な 8ms 未満を一貫維持する。

自分が試すなら

  • CustomPaint 中心の重い描画画面で、Impeller 化後のラスタライズ時間を profile モード＋DevTools で計測し直す。
  • 120Hz 端末で遷移やリスト初回スクロールの初回フレームジャンクが消えているか確認する。
  • Rive 等で描画差が出たら --no-enable-impeller で Skia と比較し原因を切り分ける。

注意点

撤去は Android 10 以上が対象。古い端末や特殊な GPU では挙動差が出うるので、対象 OS と実機での確認は引き続き必要だ。性能計測は必ず profile モードで（debug VM は大幅に遅く数値が当てにならない）。Impeller でも不要な再ビルドは別問題として残るため、const 化や Selector による局所再描画は依然有効。

ひとことまとめ

「初回のカクつき」を生んでいた実行時シェーダーコンパイルが、ビルド時 AOT で消える——これが今回の本質。Android でも Skia が消え Impeller 一本化された今、まずは自分の重い描画画面を profile で測り直すのが近道だ。
