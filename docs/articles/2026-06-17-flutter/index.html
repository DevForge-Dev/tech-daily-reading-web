2026ロードマップ「Finishing the Job」— Android Impeller完全移行とWeb WASM既定化

新機能の大盤振る舞いではなく、未完了の移行を仕上げる年。Android 10以上でレガシーSkiaを撤去しVulkanのImpellerへ一本化、WebはWASMを既定へ。GenUI/エージェント志向やFull-Stack Dartも前進する、Flutterの足場固めを読む。

📅 2026年6月17日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

Google I/O 2026 で示された Flutter の2026ロードマップは、テーマからして異色だ。題して 「Finishing the Job」——新機能を派手に増やすのではなく、これまで中途だった移行を仕上げる年だと宣言した。安定版は Flutter 3.44 / Dart 3.12、2026 も最低 4 回の安定リリースを約束する。地味に見えて、長期保守するアプリにとっては「いつ何が撤去されるか」が読めるのは大きい。

何が新しいか

最大の山場は Android の Impeller 完全移行。Android 10 以上でレガシーな Skia バックエンドを撤去し、Vulkan ベースの Impeller へ一本化する。Impeller はシェーダーをビルド時に AOT 事前コンパイルするため、初回フレームからジャンクが出にくく、iOS は Metal・モダン Android は Vulkan を直接叩いて安定120fpsを狙う。Web 側は WASM を既定へ前進。あわせて GenUI／エージェント志向アプリの探求、Full-Stack Dart の拡充、Gemini CLI / Antigravity 連携による AI 開発体験の再構築が並ぶ。デザインシステムは Material/Cupertino を material_ui/cupertino_ui へ分離、iOS/macOS は SwiftPM が既定で CocoaPods 終焉へ。Dart 3.12 は private named parameters が正式、primary constructors が実験的に入った。

足場固めの中身

状態管理の基本線は変わらず、新規は Riverpod 3.3.1 + コード生成、監査証跡の要る規制業界は Bloc 9.1.1（on<Event> ハンドラ構文に統一）、GetX は12ヶ月超のプロジェクトで非推奨。パフォーマンスはフレーム予算で考えるのが定石で、90/120Hz が主流の2026 では 1 フレーム 8〜11ms に収める。検証は必ず profile モードで行い、最頻出の問題「不要な再ビルド」は const 付与でスキップ、Consumer/Selector で局所再描画にする。

自分が試すなら

  • Android のターゲットを 10+ に揃え、Impeller 前提で実機（特にミドルレンジの Vulkan 端末）でジャンクを確認する。
  • iOS を SwiftPM へ移行し、CocoaPods 依存と Rosetta 起動を外してビルドを軽くする。
  • Web は WASM ビルドで初期表示とスクロール性能を計測し、既定化に備える。

注意点

「移行を仕上げる年」は裏を返せば撤去の年でもある。Skia 前提の独自シェーダーや Impeller で描画差の出る Rive 等は、リリース前に実機検証が要る。SwiftPM 移行も CocoaPods 前提のプラグインが残ると詰まりやすい。

ひとことまとめ

2026 の Flutter は「速くなる」より「二重だった足場を一本化して、これから速くしやすくする」年。新規は Impeller・Riverpod 3・SwiftPM を素直に前提に置くのが堅い。
