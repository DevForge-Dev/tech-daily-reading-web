clip-path と mask で作るリビール＆ワイプ — JSゼロ・RSCのまま、view() でスクロール駆動まで

画像を線で出す／ホバーでワイプ／円が広がるページ遷移。clip-path（ハードエッジ）と mask（ソフトエッジ）だけでJSほぼゼロ・Server Componentのまま実装でき、animation-timeline: view() と組めばスクロール駆動リビールもRSCのまま。使い分けと落とし穴を整理。

📅 2026年7月17日 ⏱ 約3分 🔗 出典 4 件

「画像を線でスッと出す」「ホバーで別レイヤーをワイプ」「ページ遷移で円が広がる」——こうしたリビール／ワイプ演出は、clip-path と mask の2つのネイティブCSSだけでJSほぼゼロ・Server Componentのまま実装できる。2026年に効くのは、両者がスクロール駆動アニメーション（animation-timeline: view()）と噛み合い、IntersectionObserverすら要らずRSCのまま LCP/INP/CLS に有利になる点だ。

使い分けが要点：ハードエッジ vs ソフトエッジ

clip-path はハードエッジで切り抜く（内か外かの2値）。circle()/inset()/polygon() で形を作り、その形をアニメートしてリビール・ワイプ・モーフを出す。対して mask はソフトエッジで透過（アルファ階調）。グラデーションを重ねて“ふわっと”フェード、帯状テキストリビール、破片状の消え方など、境界をぼかせるのが主戦場だ。

黄金律と落とし穴

clip-path が補間できるのは同じ図形関数どうし、かつ polygon は頂点数が一致するときだけ。噛み合わないとアニメせず瞬間切替になる。四角→星のように形を跨ぐなら全部 polygon に変換して頂点数を揃える。mask 側の要注意点は、mask-image の値そのものはアニメートできないこと——代わりに mask-position/mask-size を動かす。対応も古いWebKit/Blink向けに -webkit-mask-* の併記が安全だ。

.wipe{clip-path:inset(0 100% 0 0);transition:clip-path .6s}
.wipe:hover{clip-path:inset(0 0 0 0)}

Next.js（App Router）での組み込み

純CSSなので 'use client' は不要。ワイプもスクロールリビールも Server Component に置ける。JSが要るのはポインタ追従スポットライトくらいで、その部分だけ小さな client 島にし、再レンダーせず style.setProperty() でCSS変数を直更新して requestAnimationFrame でバッチする。next/image にも効き、寸法を予約するのでCLSは出ない。

注意点

大きなワイプや円の拡大は前庭系に負担になり得る（WCAG 2.3.3）。@media (prefers-reduced-motion: reduce) で clip-path: none/mask-image: none にし、リビール完了＝全面表示へ即落とす。意味のある情報を動きに依存させないこと。

ひとことまとめ

くっきり切るなら clip-path、ぼかすなら mask。view() と組めば JS ゼロで軽い演出になる。
