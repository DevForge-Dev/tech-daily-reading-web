CSS Motion Path — offset-path/offset-distanceで『軌道に沿う動き』をJSゼロで

path()やcircle()で軌道を定義し、offset-distanceを0→100%にアニメートするだけで要素がレール上を滑る。offset-distanceはコンポジタ(GPU)で動く数少ないプロパティで、Baseline 2022の成熟機能。純CSSなのでApp RouterのServer Componentのまま使える。

📅 2026年7月15日 ⏱ 約3分 🔗 出典 3 件

リストの並べ替えやフェードはよくあるが、曲線や円に沿った動きとなると急にleft/topやrAFでの座標計算が必要になりがちだ。これを純CSSだけで、しかもGPU合成に載せて実現するのがCSS Motion Path。まだ使っている現場が少なく、差がつく演出でもある。

何が新しい／なぜ効くか

仕組みは単純で、offset-pathで軌道（レール）を定義し、offset-distanceを0%→100%にアニメートすると要素がその上を滑る。進行方向へ頭を向けるoffset-rotate: autoがあるので、紙飛行機や彗星の尾のような演出もJSゼロ。ポイントはoffset-distanceがtransform/opacityと同格でコンポジタ（GPU）アニメート可能なこと。スクロールやrAFの座標計算を排除でき、INPに悪影響を与えない。path()やoffset-distanceはBaseline 2022で全主要ブラウザ安定、フォールバック不要だ。

App Routerでの組み込み

純CSSなのでServer Componentのまま動く。'use client'不要、hydrationゼロでCLS/INPに有利。時間ベースの常時ループはそのままanimationで回し、スクロール連動にしたいときだけanimation-timeline: scroll()/view()を併用すればこれもJS island無しで完結する。

.comet {
  offset-path: path("M 20 120 C 120 -20, 280 260, 420 60");
  offset-rotate: auto;      /* 接線方向へ頭を向ける */
  animation: run 4s linear infinite;
}
@keyframes run { to { offset-distance: 100%; } }

落とし穴

座標系は要素の含有ブロック基準で、path()の数値はレスポンシブに追従しない。可変幅で正確に沿わせるなら親を固定サイズにするか、%やcalcが使えるshape()を使う。ただしshape()は新しめなので@supportsでオプトインし、非対応はpath()に縮退させる。filter/maskを同じ要素に付けるとGPU合成から外れやすいので装飾は内側の子へ。prefers-reduced-motion: reduceではanimation: noneにしoffset-distanceを最終値へ固定する。

ひとことまとめ: 軌道に沿う動きを、成熟した純CSSで・GPUに載せて・RSCのまま。scroll-drivenと組み合わせると強い。
