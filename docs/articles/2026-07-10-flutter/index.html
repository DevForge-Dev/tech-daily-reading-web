Widget Previewerのロードマップ — 3.35実験導入から3.47 interactive previewへ

アプリ全体を起動せず単一ウィジェットをIDE内でプレビューする開発ループが本格化。3.44でVS Code/IntelliJ統合が強化され、3.47ではinteractive previewが見込まれる。material_ui分離やWeb hot reload正式化と合わせ、今日の主題は反復速度。

📅 2026年7月10日 ⏱ 約3分 🔗 出典 5 件

Flutterの開発体験で長年の不満だったのが「小さなUI変更の確認にもアプリ全体の起動が要る」こと。今日の主役Widget Previewerはそこを直接解決する機能で、進化のロードマップが見えてきた。3.35でexperimental導入→3.44でVS Code / IntelliJ（Android Studio含む）とのIDE統合を強化し、アプリ全体を実行せずに単一ウィジェットをプレビュー→3.47でinteractive previewとデバッグ強化が見込まれる。UI単体の反復速度が段階的に上がっていく。

足元の安定版はFlutter 3.44.0 / Dart 3.12.2（次の安定版は8月予定）。3.44の地殻変動はMaterial/Cupertinoの独立パッケージmaterial_ui / cupertino_uiへの移行で、3ヶ月のSDKサイクルに縛られずpub.devで随時更新される体制になった。Material 3 Expressiveの新機能もSDKリリースを待たず届く。Webのhot reloadがフラグ不要で正式利用可能になり、Web開発の反復ループがモバイルと同等になったのも大きい。DevToolsはWASMが既定となり軽量化。なお3.38以降へ上げる際はApple必須のUIScene lifecycle対応でコード移行が必要な点に注意。

アニメーション側ではRive 0.12+のnamed eventsが面白い。アニメーション側からFlutterへイベントを発火でき、ハプティクス・サウンド・ナビゲーションを特定のインタラクション時点で正確にトリガーできる。state machineがアニメロジックを描画から分離するので、デザイナーはコードに触れず反復し、実装側はboolean/number/triggerの入力を繋ぐだけ。土台のImpellerはAOTシェーダーコンパイルでジャンクを排除し、複雑シーンの平均ラスタライズ時間を約50%短縮している。

状態管理は基本線どおり新規はRiverpod 3.x既定（Mutations・オフライン永続化はexperimental、大規模アプリでProvider比20〜25%のメモリ削減）、監査証跡が要る規制業界はBloc 9.x（bloc v9.2.x / flutter_bloc v9.1.x）。今日の教訓は「選定はベンチマークでなくチーム習熟度に従う」——BLoCに習熟したチームは乗り換えるより出荷する方が速い。パフォーマンスは90/120Hz主流でフレーム予算8〜11ms、計測は必ずprofileモードで。

自分が試すなら、3.44でWidget Previewerを有効化して普段のコンポーネント修正ループに組み込み、体感の差を確かめる。注意点はexperimental段階の機能ゆえIDE・プロジェクト構成によって挙動差があり得ること。

ひとことまとめ: 「アプリを起動しないUI開発」への布石が着々と進む。3.44は静かだが、反復速度という一番効く場所を変えつつある。
