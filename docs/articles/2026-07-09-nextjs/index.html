Server Actions × useActionState — App Router フォームの定石を固める

React 19 で安定した useActionState にフォーム状態を畳み込み、JS 未ロードでも動くプログレッシブエンハンスメントと「throw しない型付きエラー契約」で堅牢なフォームを組む。

📅 2026年7月9日 ⏱ 約3分 🔗 出典 5 件

App Router で「フォーム送信 → サーバーで検証・書き込み → 結果を UI に返す」を書くと、useState で loading / error / success を手管理する定型コードが積み上がる。React 19 で安定化した useActionState はこれを1フックに畳み込み、しかも JS が未ロードでも素の HTML フォームとして送信できる（プログレッシブエンハンスメント）。変更系の定石として押さえておきたいパターンだ。

形は const [state, formAction, isPending] = useActionState(action, initialState, permalink?)。action は (previousState, formData) => newState のシグネチャで、返り値がそのまま次の state になる。第3要素の isPending で送信中が判定でき、送信ボタンの disabled 程度なら useFormStatus なしで書ける。要点は <form action={formAction}> と関数を直接渡すこと（onSubmit ではない）。これで JS 読込前でも POST が飛ぶ。

設計の核は型付きエラー契約。Server Action は実体としては公開 POST エンドポイントなので、例外を投げるのではなく、毎回同じ形のオブジェクト（例: { ok: false, fieldErrors, values }）を返す。fieldErrors でインライン表示し、values で失敗時に入力値を復元する。JS なしの送信ではクライアント state が残らないため、値の復元はサーバーから返すのが確実だ。サーバー側の手順は「① 認証（DAL の verifySession 等）→ ② Zod で検証 → ③ 書き込み → ④ revalidatePath / revalidateTag」で固定する。

const [state, formAction, isPending] = useActionState(createNote, initial);
// action 側: (prev, formData) => Promise<FormState>
// 失敗: { ok:false, fieldErrors, values } / 成功: { ok:true, message }

落とし穴は3つ。useFormStatus は親の form から状態を拾うため、form を描画するコンポーネント自身では動かず、SubmitButton を子に切り出す必要がある。useOptimistic の自動ロールバックは async 関数が throw したときだけ働くので、削除のようにロールバックしたい操作は逆に throw させる設計にする（インライン表示したい操作は throw しない、という使い分け）。そして Next.js 16 はインラインクロージャ変数を転送時に暗号化するが、これは認可モデルではない。'use server' を付けても認可は付与されず、毎回の入力検証・認証・権限チェックは自前で行う。.bind() で ID を渡す場合は prevState の後ろに引数がずれる点（React #31356）にも注意。

自分が試すなら、既存の「useState + fetch」のフォームを1つ選んで useActionState + 型付き契約に置き換え、JS を無効にしても送信・エラー表示・値復元が成立するか確認する。ひとことまとめ: フォームは「throw しない契約 + form action 直渡し」。この2点だけで堅牢さと PE が同時に手に入る。
