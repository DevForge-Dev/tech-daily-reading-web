Claude Code

「対話」と「自動化」が分かれた日 — 課金分離・サブエージェント・プラグイン検索を読み解く

6/15施行のプラン分離でAgent SDKやclaude -pがサブスク枠外に。同時にavailableModels不具合修正、ネストしたサブエージェントGA、プラグイン検索が入った。自分の使い方への影響を整理する。

📅 2026年6月16日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

2026年6月15日、Claude有料プランの利用枠が「対話（Interactive）」と「自動化（Programmatic）」に明確に分離された。これまでサブスクの枠内で回せていた claude -p や Agent SDK 経由の実行が、別枠で課金される構造になった。日々スケジュールタスクや CI で Claude Code を回している人にとっては、課金の前提が変わる地味だが重い変更だ。

何が変わったか

ポイントは「通常の対話利用は従来どおり」「自動化はサブスク枠外で別途消費」という線引きだ。つまり対面で Claude Code を叩く分にはこれまでと体感は変わらないが、プログラム的に無人で走らせる使い方は実質コスト増になりうる。あわせて直近の changelog では、ANTHROPIC_DEFAULT_*_MODEL 環境変数経由でエイリアスモデルがブロック対象モデルへ転送され得た availableModels 強制の不具合が修正された。意図せず制限対象モデルに流れる穴がふさがった形だ。

サブエージェントとプラグインの強化

機能面では ネストしたサブエージェントが一般提供になり、サブエージェントの中からさらにサブエージェントを呼べるようになった。設計の原則は「1エージェント1ジョブ」。description を広げすぎると何にでもマッチして委譲精度が落ちるため、調査・並列処理は独立文脈のサブエージェントに切り出し、検索結果やログを本流の会話に持ち込まないのが定石だ。

さらに 新しいプラグイン検索でマーケットプレイスのプラグイン（skills・hooks・subagents・MCP/LSP を1パッケージ化）を探しやすくなり、.mcp.json を手編集せずワンコマンドで導入できる。

自分が試すなら

  • スケジュールタスクや CI の claude -p 利用量を /usage のカテゴリ別内訳で棚卸しし、自動化枠の消費を把握する。
  • 重い調査系ワークフローを 1ジョブ1サブエージェントに分割し、メイン文脈の汚染とトークン浪費を抑える。
  • hooks で決定論的なガードレールをシステムレベルに固定し、モデル解釈頼みの運用をやめる。

注意点

「対話は据え置き」というメッセージに安心して自動化のコストを見落とすと、月末の請求で驚くことになる。無人実行が中心の人ほど、6/15以降は課金構造を一度確認しておきたい。

ひとことまとめ

対話と自動化の分離は、Claude Code を「手で使う道具」から「組み込むインフラ」へ位置づけ直す布石。サブエージェントのGAとプラグイン検索は、その自動化を質高く組む土台になる。
