権限と安全装置の再設計 — Tool(param:value)構文とautoモードの分類器ガード

v2.1.178でツール入力値にマッチするTool(param:value)権限ルールが登場。Agent(model:opus)で特定モデルのサブエージェント起動まで止められる。同時にautoモードがBedrock/Vertex/Foundryへ広がり、確認プロンプトを分類器の事前評価に置き換えた。無人運用の安全設計を読み解く。

📅 2026年6月17日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

自動化が進むほど「何を許可して、何を止めるか」をどれだけ細かく制御できるかが効いてくる。これまでの権限ルールは Bash や Agent といったツール単位での許可/拒否が基本で、「同じツールでも特定のパラメータのときだけ止めたい」という粒度には届かなかった。2026-06-15 公開の Claude Code v2.1.178 はここに手を入れ、ツールの入力値そのものにマッチする権限構文を導入した。スケジュールタスクや CI で無人実行している人ほど直撃するアップデートだ。

何が新しいか

目玉は Tool(param:value) 構文（* ワイルドカード対応）。ツールの入力パラメータ単位で許可/拒否を書けるようになった。たとえば Agent(model:opus) と書けば、Opus を使うサブエージェントの起動だけをブロックできる——コスト高いモデルの誤起動を防ぐ、といった制御が宣言的に書ける。あわせてワークフロー保存が最寄りの .claude/workflows/ を対象にするよう改善され、auto モードのサブエージェント起動が分類器の事前評価を通るようになった。直近では Safe Mode（設定破損時の安全フォールバック）、作業ディレクトリを移す /cd、デフォルトモデルの Opus 4.8、カテゴリ別の /usage 内訳も到着している。

autoモードの安全モデル

もう一つの軸が auto モードの拡大だ。これまでの「ツールごとに確認プロンプトを出す」方式を、バックグラウンドの安全性チェックに置き換える。AI の分類器が各ツール呼び出しを実行前に評価し、危険な操作だけをブロックする仕組みで、サードパーティの Bedrock / Vertex / Foundry（Opus 4.7／4.8 向け）でも使えるようになった。v2.1.178 ではサブエージェント起動もこの事前評価を通るので、委譲先まで含めて安全網がかかる。

自分が試すなら

  • スケジュール/CI の権限を Tool(param:value) で締め、Agent(model:opus) で高コストモデルの誤起動を封じる。
  • /usage のカテゴリ別内訳で自動化枠の消費を棚卸しする。
  • 分類器頼みにせず、決定論的に止めたい所は hooks でガードレールを併設する。

注意点

分類器ベースの安全チェックは確率的で万能ではない。bypass モードでも .git/・.claude/（commands/agents/skills 除く）・.vscode/・.husky/ など保護パスへの書き込みは確認が残る——この最後の砦は外さない設計が安全だ。

ひとことまとめ

v2.1.178 は「許可の粒度」を一段細かくし、auto モードは「確認の方法」を分類器へ寄せた。手で確認する運用から、ルールと分類器で守る無人運用へ——その移行を支える土台が揃いつつある。
