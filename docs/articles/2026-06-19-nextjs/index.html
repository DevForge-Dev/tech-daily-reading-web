Next.js 16 の next/image 破壊的変更で Core Web Vitals を底上げ — priority→preload・qualities 必須化

Core Web Vitals(LCP/CLS/INP)は Next.js では画像とフォントの扱いで大半が決まる。Next.js 16 で next/image の API が変わり、priority は preload へ改名、qualities は必須化(既定[75])。過去のコピペが古くなる破壊的変更を押さえつつ、next/image と next/font で三指標を底上げする実装手順を整理する。

なぜ今これが話題か

Core Web Vitals の三指標——LCP(表示速度)・CLS(レイアウト安定性)・INP(応答性)——は、Next.js では画像とフォントの扱いでほぼ決まる。遅いページの多くは hero 画像が LCP 要素で、Web フォントの読み込みが CLS と FOIT を生み、肥大化した JS が INP を悪化させる。next/image と next/font はこの定番対策をビルトイン化しており、正しく使うだけで「LCP < 2.5s・CLS ≈ 0」に届きやすい。

何が新しいか（Next.js 16 の破壊的変更）

公式ドキュメント(v16.2.9 / Version History の v16.0.0)で確認した要点:

  • priority prop は非推奨化され preload prop に置換。preload={true} が <head> に <link rel="preload"> を挿入する。
  • qualities が必須化。next.config の images.qualities で許可画質を明示しないと任意画質で叩かれる攻撃面になるため。既定は [75]。一致しない値は近い値へ丸められ、REST API 直叩きは 400 を返す。
  • セルフホスト向けに maximumDiskCacheSize(16.1.7) などを追加。dangerouslyAllowSVG は依然デフォルト無効。

仕組みのポイント

LCP：hero には preload を付け <head> から先読みさせ、loading="lazy" は厳禁。ビューポートで LCP 要素が変わるレスポンシブでは preload ではなく fetchPriority="high" が意図明確。sizes を正しく書くとフル srcset が出て、小型端末が巨大画像を落とさずに済む。

CLS：画像は width/height か静的 import か fill でアスペクト比の枠を予約。フォントは next/font が size-adjust/ascent-override 等を自動注入して差し替え時のズレを潰す(既定 display:'swap' で FOIT 回避＆セルフホスト化)。INP：RSC でクライアント JS を削り、Client Component は対話が要る「島」に限定する。

自分が試すなら

// next.config.js — 16では必須
images: {
  qualities: [50, 75, 100],
  formats: ['image/avif', 'image/webp'],
}

hero は静的 import + preload + placeholder="blur"、style={{width:'100%',height:'auto'}} で CLS を回避。AVIF は WebP より約20%小さいがエンコードは約50%遅い(初回のみ)ので、多くの用途では WebP も併走させる。

注意点

過去記事の priority コピペは 16 で警告になる。qualities 未設定だと意図せず既定 [75] に縛られ、quality={90} が丸められる。セルフホスト＋前段 CDN では Accept ヘッダを転送しないと WebP/AVIF 判定が効かない。最適化画像は無効化機構が無いので minimumCacheTTL は低めにするか src を変える運用に。

ひとことまとめ

Next.js 16 では priority→preload、qualities 必須化がコア変更。hero に preload、その他は遅延、qualities を明示、フォントは next/font に任せる——この4点で LCP/CLS は自動的にかなり良くなる。
