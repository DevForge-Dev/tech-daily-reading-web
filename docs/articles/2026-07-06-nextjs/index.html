RSCを“もう一つのローダー”に — TanStack Query v5をApp Routerで

App RouterのサーバーデータはRSC＋fetchで足りる場面が多い。だが再取得・楽観的更新・無限スクロール等の“インタラクティブなサーバー状態”はTanStack Query v5が最有力。正解はRSCでprefetch→dehydrate→クライアントのHydrationBoundaryで復元。最大の落とし穴はstaleTime:0で、prefetchが即二重取得になる点だ。

📅 2026年7月6日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

App Routerではサーバー状態の取得はRSC＋fetch＋Server Actionsでほぼ完結する。しかし再取得・キャッシュ無効化・楽観的更新・window focus refetch・無限スクロール・ポーリングが要るダッシュボードやフィードでは、依然TanStack Query v5（旧React Query）が最有力だ。前日扱ったZustand（クライアント状態）とは守備範囲が違い、こちらはサーバー由来の非同期データを担当する。

何が新しい/重要か

App Router時代の正解は、RSCを“もう一つのローダー”として使い、サーバーでprefetch→dehydrateでキャッシュを直列化→Client側の<HydrationBoundary>で復元する形。初回描画はprefetch済みデータで即埋まり、以降はクライアントのQueryが引き継ぐ。QueryClientは「サーバーはリクエスト毎に新規／ブラウザはシングルトン」が鉄則で、グローバル1個はクロスリクエスト汚染・hydration mismatchを招く。サーバー判定は現行ドキュメントのenvironmentManager.isServer()を使う。

最大の落とし穴：staleTime

ストリーミングと非推奨パターン

v5.40.0+ではshouldDehydrateQueryを拡張してpending Queryもdehydrateでき、awaitせずprefetchを撃って解決順にクライアントへストリーミングできる（消費側はuseSuspenseQuery）。逆にServer ActionsをqueryFnにしない——直列実行でpendingのまま固着する。Server ActionsはuseMutation専用、データ取得はRoute Handler/tRPCで、が公式スタンスだ。

自分が試すなら

makeQueryClient()＋getQueryClient()を用意しstaleTime:60_000を既定に。prefetch前提でない画面はuseSuspenseQueryではなく壊れにくいuseQueryを選ぶ。RSCのfetchQuery戻り値を描画に使わず、所有をどちらか一方に寄せて同期ずれを防ぐ。

ひとことまとめ

RSCをローダー化してprefetch→dehydrate→HydrationBoundary。staleTimeを上げ、Server Actionsはmutation専用。新規は標準(RSC+fetch)で始め、必要になってから足すのが公式の勧めだ。
