ライブラリを足す前の第一選択肢 — Web Animations API（element.animate()）を App Router で使う

element.animate() は2020年3月からBaselineで安定。CSSでは無理な動的値・play/pause/reverse・seek・getAnimations() 一括操作が“0バイト”で手に入る。終了状態は fill:'forwards' でなく commitStyles()、Animation は state でなく ref に退避が要点。

📅 2026年7月18日 ⏱ 約3分 🔗 出典 4 件

Motion や GSAP を入れる前に、まず検討すべきはブラウザ標準の Web Animations API（WAAPI）だ。element.animate(keyframes, options) を呼ぶだけで CSS @keyframes 相当をJSから生成・再生・停止・反転・シークできる。CSS Animations/Transitions は内部でこのAPIの上に実装され、コンポジタ（GPU）駆動で requestAnimationFrame 手回しより速い。element.animate() は2020年3月から全モダンブラウザで安定（Baseline）でフォールバック不要、Motion の useAnimate や AutoAnimate の内部もこれを薄くラップしているだけ——バンドルに1バイトも足さずに済むのが最大の魅力だ。

CSSでは届かない領域

宣言的CSSが苦手な動的な値・実行時制御——データ由来の duration、ユーザー操作での play()/pause()/reverse()、スクロール量に応じた currentTime スクラブ、document.getAnimations() でサイト全体を一括減速（vestibular配慮）——が素の言語機能でできる。anim.finished（Promise）で await による連鎖も書きやすい。ただし用語がCSSと違う：ms単位（3sでなく3000）、iterations:Infinity、easing、そして既定 easing は linear。

App Router での組み込み

WAAPIはDOM API＝ブラウザ専用。だから'use client' 必須、useRef で要素参照、Animation インスタンスも ref に退避（stateに載せると毎フレーム再レンダー地獄）。useEffect 内で el.animate(...) を呼び、クリーンアップで anim.cancel()——Strict Mode の二重実行もこれで吸収。SSRでは走らないので hydration mismatch は起きず、初期表示は静的に返しマウント後にアニメで強化するプログレッシブエンハンスメント設計にする。prefers-reduced-motion は matchMedia で判定し、動きを消しつつ finish() で最終状態だけ残す。

注意点

width/height/top/left の animate は毎フレーム再レイアウトで重い——transform/opacity に寄せる。単一キーフレームは throw しうるので必ず2キー以上。commitStyles() は要素が可視でないと throw。JS版 ScrollTimeline/ViewTimeline は現状Chromium系のみで、宣言的でよいならCSSの animation-timeline が無難だ。

ひとことまとめ

命令的コントロールが要るだけなら、ライブラリの前にWAAPI直で足りることが多い。ref退避・cancel掃除・commitStyles、この3点を押さえれば0バイトで戦える。
