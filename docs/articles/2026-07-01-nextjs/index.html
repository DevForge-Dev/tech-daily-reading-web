Turbopack が Next.js 16 の既定に — dev も build も無設定で2〜5倍、FSキャッシュとメモリ90%減まで

Next.js 16でTurbopackがdev/buildの両方で安定版・既定バンドラになった。本番ビルドは2〜5倍、Fast Refreshは最大10倍。設定キーはトップレベルturbopackへ昇格し、webpackは--webpackでオプトイン側へ回った。ファイルシステムキャッシュとメモリエビクションで、ウォームビルドとdevメモリも大幅改善。移行の落とし穴も押さえておく。

📅 2026年7月1日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

これまでTurbopackは「devだけ速いツール」だった。それがNext.js 16でdevとbuildの両方の既定バンドラに昇格し、next build（本番ビルド）まで担うようになった。歴史はnext devがv15.0で安定、next buildが15.3で実験→15.5でbeta→16.0で既定。webpackは--webpackでのオプトイン側に回った。すでにNext.js 15.3+では開発セッションの50%超・本番ビルドの20%超がTurbopack稼働で、vercel.comやnextjs.org自身もTurbopackビルドだ。

何が新しいか

新規も既存も基本は無設定でnext dev/next buildがそのままTurbopack。設定キーはexperimental.turbopackからトップレベルのturbopackへ昇格した。効果は本番ビルド2〜5倍、Fast Refresh最大10倍。さらに16.1〜16.3で育ったFSキャッシュにより、ウォームビルドが大幅高速化し、devメモリは最大90%削減される。

仕組みのポイント

速さの源は3つ。client/serverを単一グラフで扱うUnified Graph、一度やった仕事を再計算しない関数レベルのインクリメンタル計算＋並列化（コンパイル時間はルートの大きさでなく変更の大きさに比例）、devサーバが要求した分だけ束ねるLazy Bundlingだ。CSSはLightning CSS、JS/TSはSWCが処理し、型チェックはしないのでtsc --watchやIDEに任せる。

自分が試すなら

  • 既存アプリは移行前に一度next buildを回し、webpackプラグイン・webpack()設定・Sassの~・CSS Module順序依存の有無を確認する。
  • CIではexperimental.turbopackFileSystemCacheForBuild: trueを有効化し、.nextを実行間で持ち越してウォーム化する。
  • 詰まったら当面next build --webpackで逃げられる。
// next.config.js — 設定はトップレベル turbopack キー
module.exports = {
  turbopack: { resolveAlias: { '~*': '*' } }, // Sass の ~ 構文救済
  experimental: { turbopackFileSystemCacheForBuild: true },
}

注意点

webpackプラグインは非対応（loaderは対応）、webpack()設定は無視、Sassの~構文や一部レガシーCSS Modules機能も非対応。カスタムwebpack設定が残っているとnext buildは誤設定として止まる。意図的にwebpackを使うなら明示的に--webpackを付ける。

ひとことまとめ

Next.js 16はTurbopackを「本番の既定」に据えた。無設定で2〜5倍、FSキャッシュでウォームはさらに速く、devメモリは9割減。移行は事前の一度のbuildで落とし穴を洗い出すのが近道だ。
