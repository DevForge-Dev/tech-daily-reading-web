Lenisで慣性スムーススクロール（App Router）——ネイティブscrollTopを毎フレーム動かす設計

Lenis 1.3.x は自前のrAFループで「ネイティブのscrollTopを毎フレームlerpで更新」する設計。だからCSS scroll-driven animations・position:sticky・IntersectionObserver・GSAP ScrollTriggerがそのまま連動する。App Routerではプロバイダだけ use client にして配下のRSCは保つ。旧 @studio-freight 系は非推奨、現行は素の lenis（+ lenis/react）。

📅 2026年7月19日 ⏱ 約3分 🔗 出典 4 件

受賞サイト系の「ぬるっと動く」体感の定番、Lenis（darkroom.engineering製）を Next.js App Router に正しく組み込む話。ネイティブの scroll-behavior: smooth がアンカー移動時だけなのに対し、Lenisはホイール/タッチ操作そのものに慣性（lerp）を効かせて追従させる。GitHub 13.9k★。

何が新しい/重要か

核心は設計思想だ。Lenisは自前のrAFループでネイティブの scrollTop を毎フレーム更新する。locomotive-scroll 等の旧世代がコンテンツを transform でズラしていたのと決定的に違い、本物のスクロール位置が動く。結果、CSS scroll-driven animations（animation-timeline: scroll()/view()）・position:sticky・IntersectionObserver・GSAP ScrollTrigger がそのまま連動する。「一つのスクロール駆動源」に複数演出を全部ぶら下げられるのが強い。

仕組みのポイント

virtual-scroll で wheel/touch を横取り→ targetScroll を更新→毎rAFで lerp（既定0.1）で寄せてネイティブ位置に反映。慣性の芯はこれだけだ：

let current=0, target=0; const LERP=0.1
addEventListener('wheel', e=>{ e.preventDefault(); target+=e.deltaY }, {passive:false})
function raf(){ current += (target-current)*LERP; scrollTo(0,current); requestAnimationFrame(raf) }

App Routerでは Lenis が window に触るためServer Component不可。プロバイダだけ 'use client' にし、children で RSC を通せば配下は client 化しない。初期化は useEffect（ブラウザのみ）で走るのでhydration mismatch は出ず、JS前でも素のスクロールで読めるプログレッシブエンハンスメント。

GSAPと同期／自分が試すなら

複数rAFが走るとScrollTriggerがカクつくので、autoRaf:false で gsap.ticker.add(t=>lenis.raf(t*1000)) と lenis.on('scroll', ScrollTrigger.update) で1ループに束ねる。まずは公式アダプタ lenis/react の <ReactLenis root> ＋必須CSS import 'lenis/dist/lenis.css' から始めるのが最短。

注意点

CSS scroll-snap は非対応（要るなら lenis/snap）。Safariは60fps上限、iframe上ではホイールが伝播しない。options を毎レンダー生成して依存に入れると「Maximum update depth exceeded」——useRef退避＋空依存で初期化。そしてアクセシビリティ上の是非があるので prefers-reduced-motion を尊重し、モバイルは既定の syncTouch:false のままが無難。

ひとことまとめ

「ネイティブscrollTopを動かす」一点で、Lenisは既存のスクロール演出資産をそのまま活かせる。加点演出として reduced-motion を必ず添えて。
