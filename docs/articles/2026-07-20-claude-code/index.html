v2.1.212——暴走ループを止めるセッション上限、/fork のバックグラウンド化、/resume ピッカー

WebSearch/サブエージェント呼び出しにセッション単位の上限（各既定200）を新設し暴走ループを抑止。/fork は会話をバックグラウンドセッションへ複製、その場実行は /subtask に分離。2分超の MCP 呼び出しは自動でバックグラウンド化、/resume は削除済みを含む過去セッションのピッカーに。plan モードの無確認ファイル変更と worktree のシンボリックリンク脱出も修正。

📅 2026年7月20日 ⏱ 約2分 🔗 出典 3 件

この日のClaude Codeは暴走対策と安定化が主役だ。v2.1.212が公開され、v2.1.207〜212に連なる安全性・信頼性強化の総仕上げとして、エージェントが延々と走り続ける事故を構造的に止めにきた。号令一下で数百のサブエージェントやWeb検索が連鎖しうる時代だからこそ、上限とバックグラウンド化の整備が要になる。

何が新しいか

目玉はセッション単位の上限。1セッションあたりのWebSearch呼び出し（既定200・CLAUDE_CODE_MAX_WEB_SEARCHES_PER_SESSIONで調整）とサブエージェント起動数（既定200・CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSIONで調整）に天井が付き、暴走ループを抑止する。/forkは会話を新しいバックグラウンドセッションへコピーする方式に変わり、従来その場で起動していたサブエージェントは/subtaskへ分離。2分を超えるMCPツール呼び出しは自動でバックグラウンドへ移り、/resumeは削除済みを含む過去セッションのピッカーを開く。

地味だが効く修正

安全面では、planモードが権限確認なしにファイル変更用のBashを自動実行してしまう不具合と、シンボリックリンク経由でworktreeがリポジトリ外にファイルを作る不具合が修正された。前者は「planモードは読むだけ」という信頼を損なう類のバグで、修正の価値が高い。

自分が試すなら

定期タスクや横断調査でサブエージェント・WebSearchを多用する運用なら、まず既定200のままで十分な安全網になる。逆に大規模な一括処理で上限に当たるなら環境変数で引き上げる。長時間のMCP処理が自動でバックグラウンド化されるので、待ちの体感も改善する。

注意点

上限はあくまで暴走の保険で、コスト管理の代替ではない。/forkがバックグラウンド化した分、/resumeピッカーで「どこで何が走っているか」を把握する習慣を持たないと取り回しを見失いやすい。

ひとことまとめ

「上限・バックグラウンド・ピッカーで、暴走を止めつつ多重実行を御す」アップデート。planモードの無確認実行修正も地味に効く。
