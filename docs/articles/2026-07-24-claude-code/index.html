v2.1.215——/verify・/code-review を「手動起動限定」に変更、7月は fork のバックグラウンド化・FS隔離の独立制御まで

7/24 の v2.1.215 は、これまで状況に応じて自動発火していた /verify と /code-review を、打ったときだけ走る手動起動限定に切り替えた。重い長時間チェックのコストを予期せず払わされないための制御性強化だ。7月通してみると /fork のバックグラウンドセッション化・ファイルシステム隔離の独立制御・スクリーンリーダー対応など、レビュー動線と信頼性の底上げが軸になっている。

📅 2026年7月24日 ⏱ 約2分 🔗 出典 3 件

背景から。エージェントに号令一下でコードを書かせる使い方が普通になり、生成された差分の検証(/verify)とレビュー(/code-review)をどう回すかが日々の運用課題になっていた。この2つは実行時間もトークン消費も大きい「重いチェック」で、勝手に走ると予期せぬコストになる。v2.1.215はそこに手を入れた回だ。

何が新しいか

核心は、/verify と /code-review を自動実行から外し、ユーザーが明示的に打ったときだけ走る手動起動限定にしたこと。バンドルされた他のスキルは従来どおり関連時に自動起動されるが、この重い2つだけは発火タイミングの主導権を利用者に戻した。「気づいたら長時間チェックが走っていた」を避けつつ、必要なときは /verify / /code-review と打てばよい。

7月全体の潮流

月間で見ると、/fork が会話を新しいバックグラウンドセッション（claude agents に独立した行として現れる）へ複製する挙動に変わり、旧来の「セッション内サブエージェント起動」は /subtask に分離された。v2.1.216 ではファイルシステム隔離を独立して制御でき、コード実行の柔軟性が上がった。v2.1.208 ではスクリーンリーダー対応や vim インサートのリマップ（jj で Esc）などアクセシビリティ改善が入り、折りたたみツール要約に経過時間のライブカウンタも付いた。全体として review ワークフロー・アクセシビリティ・信頼性の底上げが今月の軸だ。

実務での使いどころ

実務ガイドは口を揃えて「効くのは良いプロンプトより良いハーネス」と言う。ハーネスとは、引き締まったメモリ(CLAUDE.md)、実効的な権限ポリシー、スキルのライブラリ、譲れない一線を強制する少数の hooks、そして「広く探索するなら fork・深く推論するなら inline」という判断の総体。何でもフォークするのは失敗モードで、推論の重い作業は inline に残す。ある30名の開発組織は共有スキル22個＋結線済み hooks 11個を整え、4か月で35%の生産性向上を持続したという。

注意点・ひとことまとめ

注意は、これまで自動 /verify・/code-review に頼っていた人が「レビューが走らなくなった」と感じる点——今後は自分で打つ運用に切り替える。まとめると「重いチェックは手動に、探索は fork・深掘りは inline、成果はハーネスで積む」。派手さより制御性の回だ。
