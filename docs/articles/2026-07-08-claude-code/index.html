Sonnet 5 が既定に、v2.1.202 は“動的ワークフローの規模”を手綱に

7/8のClaude Codeは二本立て。既定モデルが Sonnet 5（ネイティブ1Mトークン、8/31まで$2/$10）へ切り替わり、長大コードベースをサブエージェント無しで一気に読める余地が広がった。もう一方の v2.1.202 は /config に「Dynamic workflow size」を追加し、Claudeが生成するエージェント数を small/medium/large で制御可能に。OTel属性 workflow.run_id/name で1回の実行を再構成できるようにもなった。

📅 2026年7月8日 ⏱ 約3分 🔗 出典 3 件

なぜ今これが話題か

動的ワークフローで数十〜数百のサブエージェントを一括ファンアウトできるようになった反面、「勝手に大量のエージェントが立ち上がってトークンと時間を食う」という制御性の不安が残っていた。7/8のアップデートは、モデルの地力を底上げ（Sonnet 5）しつつ、ワークフローの規模と可観測性を人が握れるようにする（v2.1.202）という、攻めと手綱の両面を同時に入れてきた。

何が新しいか

公式リリースノートによると、Claude Sonnet 5 が Claude Code の既定モデルになった。ネイティブで100万トークンのコンテキストを持ち、8月31日まで $2/$10 per Mtok のプロモ価格、v2.1.197 以降でアクセスできる。Sonnet 4.6比で推論・ツール利用・コーディングが大きく伸び、同社史上もっともエージェント的なSonnetと位置づけられる。もう一方の v2.1.202（計18件）は /config に 「Dynamic workflow size」 を追加し、動的ワークフロー時に生成するエージェント数を small / medium / large の規模感で制御できる。さらにワークフロー生成エージェントのテレメトリに workflow.run_id と workflow.name のOpenTelemetry属性が付き、OTelデータから1回の実行の活動を丸ごと再構成できる。アイドルなバックグラウンドシェルがメモリ圧迫時に自動回収されるようにもなった。

仕組みのポイント

1Mコンテキストの効き所は「サブエージェントに探索を分けなくても、メインの文脈に必要なファイル群をまとめて載せられる」点にある。読み込み設計をシンプルにでき、要約の往復で情報が痩せるリスクを減らせる。一方 Dynamic workflow size は、リードエージェントがファンアウトする規模を先に宣言する発想だ。small なら少数精鋭で速く安く、large なら網羅性重視で広く展開、という具合にコストと網羅性のトレードオフを事前に選ぶ。OTel属性はその実行を後から監査するための背骨になる。

# /config で規模を指定するイメージ
Dynamic workflow size: medium   # small / medium / large
# 生成エージェントのテレメトリに付く属性
workflow.run_id = "wf_2026..."
workflow.name   = "refactor-auth"

実務での使いどころ・自分が試すなら

まず既定を Sonnet 5 のまま、大きめのリポジトリをサブエージェント無しで読ませ、1Mコンテキストでどこまで一括理解できるか感触を取る。動的ワークフローを使う定型（横断リファクタ、監査スキャン）では規模を small→medium と上げてコスト差を比較。OTelを流している環境なら workflow.run_id でGrafana等にワークフロー単位のダッシュボードを作ると、暴走の早期検知に効く。

注意点・落とし穴

1Mコンテキストは“載る”からといって“毎回全部載せる”のは高くつく。必要な範囲を選ぶ設計は引き続き重要だ。プロモ価格は8/31まで——恒常価格でのコスト試算も並行して持っておくと予算がぶれない。large を安易に選ぶとエージェント数が膨らみトークンを消費するため、まずは medium 基準が無難。

ひとことまとめ

既定はSonnet 5・ネイティブ1M。ワークフローは /config で規模を選び、OTelで実行を追える。地力が上がった今こそ「載せすぎない」読み込み設計と規模の選択が効く。
