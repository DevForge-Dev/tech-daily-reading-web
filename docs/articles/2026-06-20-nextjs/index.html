Next.js App Router で FLIP レイアウトアニメ — 並び替え・展開を 60fps で滑らかに

リストの並び替え、カードの展開、グリッド⇄リスト切替——レイアウトそのものが変わる動きは CSS の transition では繋げられない。鍵は FLIP（First/Last/Invert/Play）。位置の差分を transform で打ち消してから 0 へ流すので GPU 合成で 60fps を保てる。App Router での 'use client' 境界や Motion/AutoAnimate の使い分けまで実装目線で。

なぜ今これが話題か

リストの並び替え、要素の追加/削除、カードの展開、グリッド⇄リスト切替。これらはレイアウトそのものが変わるアニメで、width/top 等を直接トランジションすると毎フレーム再レイアウトが走って重い。そもそも Flexbox/Grid の自動配置による位置変化は CSS トランジションの対象にすらならない。ここで効くのが FLIP（First・Last・Invert・Play）だ。

FLIP とは何か

変化の前と後の位置を測り、その差分を transform で打ち消してから 0 へアニメする。動かすのは GPU 合成可能な transform/opacity だけなので 60fps を保てる。Motion（旧 Framer Motion）の layout プロップ、AutoAnimate、Svelte の animate:flip も内部はすべてこの FLIP だ。

仕組みのポイント

キモは、重い処理＝2回の getBoundingClientRect() 測定（強制同期レイアウト）を最初に固めること。以降のフレームは transform 補間だけになり、ブラウザが GPU で滑らかに描く。transform は他要素のレイアウトに影響しないので、リスト全項目を同時に動かしても破綻しない。

// Last測定後：差分をtransformで打ち消して0へ（WAAPI）
const dx = first.left - last.left, dy = first.top - last.top;
if (dx || dy) el.animate(
  [{transform:`translate(${dx}px,${dy}px)`}, {transform:'none'}],
  {duration:350, easing:'cubic-bezier(.2,.8,.2,1)'});

App Router での組み込み

FLIP は DOM 測定と useLayoutEffect を使うため対象は 'use client' 必須。データは Server Component で取得し、<AnimatedList items={...} /> のようにprops でクライアント子へ渡すと、サーバの利点を保ったまま動きだけクライアントに閉じ込められる。実装は三択：バンドルを増やしたくないなら依存ゼロの自前フック、本格的なら Motion の <motion.li layout>＋AnimatePresence、最小コストなら AutoAnimate（親 ref に useAutoAnimate() を付けるだけ）。

注意点

歪み：サイズ変化を scale で表すと中の文字が伸縮するので子に逆スケール補正が要る（Motion は子にも layout で自動補正）。測定タイミング：React は useLayoutEffect で Last を測りペイント前に Invert を当てる（useEffect だと一瞬最終位置が見える）。key：インデックス key は並び替えで別物扱いになりアニメが壊れる。prefers-reduced-motion では大きな移動演出だけ消す。

ルート間の共有要素なら

ページ遷移を跨ぐ共有要素は FLIP より View Transitions API（view-transition-name）や Motion の layoutId が適任。SPA 内のレイアウト変化なら document.startViewTransition() がブラウザ内部で FLIP 相当をやってくれる（Chromium系/Safari 18+）。

ひとことまとめ

レイアウトの動きは「測って・差分を transform で打ち消して・0 へ流す」だけ。まずは AutoAnimate でリストに付けて体感し、細かな制御が要れば Motion か自前フックへ——が入りやすい。
