App Router 最強のモーダル — Parallel Routes × Intercepting Routes でURL駆動、クリックは重ねて表示・直リンクはフルページ

従来モーダルの宿痾——共有不可・リロードで消える・戻る/進むで履歴が壊れる——を一掃するのが Parallel Routes と Intercepting Routes の組み合わせ。@modal スロットと (.) 記法で、同じURLが「クリックではモーダル」「直リンク/リロードではフルページ」に切り替わる。Next.js 16.2 でも安定機能として現役。最大の落とし穴はセグメントの数え方だ。

📅 2026年6月22日 ⏱ 約3分 🔗 出典 3 件

なぜ今これが話題か

従来のモーダルには宿痾があった。URLを共有できない、リロードで消える、ブラウザの戻る/進むで履歴が壊れる。App Router の Parallel Routes と Intercepting Routes を組み合わせると、これらを一掃できる。ギャラリーの写真拡大、ログイン、カート、商品クイックビューに効き、Next.js 16.2 でも安定機能として現役だ。

何が新しい/重要か

@modal のような名前付きスロット（Parallel Routes）はURLに影響せず、親レイアウトに props として渡る。(.)/(..)/(...) の Intercepting Routes は、<Link> クリック（ソフトナビ）では別ルートを現在のレイアウトに差し込んでモーダル表示し、直リンク/リロード（ハードナビ）では本来のフルページを出す。つまり同じURLが文脈で二通りに表示される。

仕組みと最大の落とし穴

キモは「(..) はファイル階層ではなくルートセグメントを数える」点。スロット（@modal）やルートグループ（(group)）はセグメントとして数えない。だから app/@modal/(.)photo/ から app/photo/ を捕まえる時、ファイル上は2階層上でもセグメント的には同一階層なので、正しい記法は (.)。ここが最大の混乱要因だ。スロットには default.tsx（null を返す）が必須で、無いとハードナビで404になる。

app/
  layout.tsx                 // children と modal を並べる
  @modal/
    default.tsx              // return null（必須）
    (.)photo/[id]/page.tsx   // Modal で包む（インターセプト先）
  photo/[id]/page.tsx        // 直リンク時のフルページ

自分が試すなら

  • 画像ギャラリーで「クリック→モーダル、直リンク→フルページ」を実装する。
  • 閉じる操作は router.back() に統一し、戻る/進むを自然に効かせる。
  • 他ページ遷移でモーダルが残る場合は @modal/[...catchAll]/page.tsx で null を返すキャッチオールスロットを置く。

注意点

同一セグメント階層のスロットは静的/動的を混在できない（1つが動的なら全スロットが動的化）。インターセプト用 (.)photo とフルページ photo は同じ表示になるよう <Photo> を共有して齟齬を防ぐ。構成は複雑化するので、独立した loading/error やモーダルが不要ならネストレイアウトの方が単純だ。

ひとことまとめ

「クリックは重ねて、直リンクはフルページ」を同じURLで両立できるのがこのパターンの真価。まずは default.tsx と (.) のセグメント数えから押さえるのが近道だ。
