Flutter の主役『The Great Thread Merge』— Dartがプラットフォームスレッドへ、FFIで直接Swift/Kotlinを呼ぶ

Flutter 3.44 / Dart 3.12.2 の目玉は、DartをUI専用スレッドからネイティブのプラットフォームスレッドへ統合するThe Great Thread Merge。async platform channel無しにFFI経由でSwift/Kotlin APIを直接呼べるようになり、ネイティブ通信が高速・クリーンになる。Impellerの既定化、Riverpod 3.x、Material/Cupertinoの独立バージョニングも合わせて読む。

📅 2026年7月1日 ⏱ 約3分 🔗 出典 4 件

なぜ今これが話題か

安定版はFlutter 3.44.0 / Dart 3.12.2（Google I/O 2026発表）。次の安定版は2026年8月予定だ。今日の主役はThe Great Thread Merge（UIスレッド統合）。Dartの実行をこれまでの専用UIスレッドからネイティブのプラットフォームスレッドへ移し、Dartとネイティブ(Swift/Kotlin)の境界を解消する。長年オーバーヘッドの原因だった非同期のやり取りに、構造的なメスが入った。

何が新しいか

統合により、async platform channel無しでFFI経由で直接Swift/Kotlin APIを呼べるようになり、ネイティブ通信が高速・クリーンになる。周辺も動いた。Material/CupertinoはコアSDKで凍結され、将来material_ui/cupertino_uiとして独立バージョニングへ移行——デザインシステムがSDKリリースを待たず独自cadenceで更新できる。Dart 3.12は.valueのdot shorthandsを正式サポートし、dart compile wasmがstableへ。採用面では実車の2026 Toyota RAV4のダッシュボードがFlutter UI、LG webOS SDK for Flutterも近日対応する。

仕組みのポイント

描画側ではImpellerがもはや実験的でなく、iOS・モダンAndroid(API29+)の既定レンダラだ。シェーダーをビルド時(AOT)にコンパイルするためシェーダーコンパイルジャンクを解消し、アニメ起動時には命令が既にGPU待機状態——複雑アニメでも120fpsを維持する。DevToolsもWASMが既定レンダラになり、巨大ウィジェットツリーのスクロールやTimelineが滑らかになった。

自分が試すなら

  • 新規はRiverpod 3.3.1＋コード生成を既定に。監査証跡や大規模チームの厳格さが要るならBloc 9系を選ぶ。
  • ネイティブ連携がある機能は、Thread Merge前提でplatform channelをFFI直呼びに置き換えられないか検討する。
  • Impeller環境での実機検証を必ず行う（特にRive等の描画）。

注意点

ImpellerがSkiaを置換し全プラットフォーム既定へ向かう過程で、Rive Flutterランタイムに描画/性能差が出る場合がある。低スペック端末を含む複数端末でのImpeller環境の実機検証を怠らないこと。

ひとことまとめ

3.44の主役はThread Merge——Dartをプラットフォームスレッドへ統合し、FFIでネイティブを直接呼ぶ。Impeller既定化とRiverpod 3.xも合わせ、「速さ」と「ネイティブ距離の近さ」が同時に前進した。
