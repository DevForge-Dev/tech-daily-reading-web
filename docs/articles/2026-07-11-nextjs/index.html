React Compiler v1をNext.jsで — reactCompiler: trueで始める自動メモ化とRust版の現在地

手書きのuseMemo/useCallbackをビルド時の自動メモ化で置き換えるReact Compilerが実用段階に。Next.js 16はトップレベル設定で安定サポート、Rust移植でビルド40%以上高速化も視野に入った。

📅 2026年7月11日 ⏱ 約3分 🔗 出典 4 件

React Compilerは、useMemo/useCallback/React.memoを手書きしなくても、ビルド時にコンポーネントとフックを自動メモ化する最適化コンパイラだ。2025年10月7日にv1.0が安定版となり、Next.jsは16.0からトップレベル設定reactCompiler: trueで安定サポート（15系はexperimental.reactCompiler）。Metaの本番アプリで初期ロード/ページ間ナビゲーション最大12%改善、特定インタラクション2.5倍以上高速化・メモリ中立という報告があり、再レンダー削減はINP改善に直結するため「まず有効化を検討すべき」段階に入った。

仕組みとしてはBabelプラグインだが、内部ではASTを制御フローグラフベースの中間表現（HIR）に下ろし、データフローと可変性を解析してレンダーに使われる値を粒度細かくメモ化する。早期returnの後の値や条件付きメモ化、a?.bを依存として扱うなど、手書きでは書けない位置のメモ化ができるのが強み。同時にRules of Reactの検証パスを持ち、eslint-plugin-react-hooksのrecommendedにコンパイラ由来のlintが統合済みだ（旧eslint-plugin-react-compilerは削除してよい）。

// next.config.ts（Next.js 16）
import type { NextConfig } from 'next'
const nextConfig: NextConfig = {
  reactCompiler: true,
  // 段階導入: { compilationMode: 'annotation' } + 対象に 'use memo'
}
export default nextConfig

最大の落とし穴はこの「サイレントスキップ」だ。レンダー中のprops変異など違反のあるコンポーネントは壊れる代わりに黙って最適化から外れ、「動くが速くならない」新種のパフォーマンスバグになる。React DevTools（v5+）の「Memo ✨」バッジで最適化の有無を確認する。既存のuseMemo/useCallbackは急いで剥がさない — 特にeffectの依存に使う値は、コンパイラのメモ化粒度変更でuseEffectの発火頻度が変わりうるため、手書きを残すのが公式推奨。バージョンは--save-exactで固定する。Server Componentsに再レンダーは無いので、恩恵は'use client'の島に集中する点も設計上は変わらない。

ビルド速度の懸念にも答えが出つつある。Rust移植版がNext.js 16.3のTurbopackに実験的統合され、直接リンクした計測ではコンパイル全体が40%以上高速化、16.4で本組み込みが進む。自分が試すなら、新規はコンパイラ有効テンプレート＋lint最新化で全体有効、既存アプリは頻出クライアントコンポーネントにバッジが付くかの確認から始める。

ひとことまとめ: 「ビルドが遅くなるから見送り」という理由が消えつつある今、reactCompiler: trueは新規Next.jsの既定に置いてよい。
