Route Handlers を Next.js 16 で使う——HTTPの表玄関、Server Actions との住み分け

app/route.ts に GET/POST を export するだけで Web標準の Request/Response を扱う公式手段が Route Handlers。v15 で GET が既定 static→dynamic に変わり params は Promise 化、v16 の Cache Components 下では GET が UIルートと同じ prerender モデルに乗る。内部ミューテーションは Server Actions、外部が叩く/HTTP制御が要る面は Route Handler が実務の目安。

📅 2026年7月21日 ⏱ 約3分 🔗 出典 3 件

この日のNext.jsテーマはRoute Handlers——App RouterでHTTPエンドポイント（REST/Webhook/SSE等）を作る公式手段だ。app配下のroute.tsにGET/POSTなどをexportするだけで、Web標準のRequest/Responseをそのまま扱える。Pages Routerのpages/apiの後継で、両者を併用せずappに寄せるのが今の型だ。旧記事の書き方が通用しなくなる破壊的変更が重なったため、あらためて整理する価値が高い。

何が変わったか（破壊的変更）

3つの変更が効く。(1) GETの既定がstatic→dynamic（未キャッシュ）に変わった。(2) context.paramsがPromise化しawait必須になった。(3) v16のCache Components（use cache）下ではGET Route HandlerがUIルートと同じprerenderモデルに乗る。キャッシュできるのはGETのみで、POST/PUT/DELETEを1つでもexportするとルートは動的化する。GETを静的化するにはexport const dynamic = 'force-static'かexport const revalidate = 60を使う。

仕組みのポイント

第1引数はNextRequest、動的セグメントは第2引数contextでconst { id } = await ctx.params。生HTTPの利点としてnew Response(body, { status, headers })で任意ステータス・Content-Typeが自由になる（Server Actionsにはできない領域）。SSEやLLMトークン逐次配信はReadableStreamをそのままreturnでき、Webhookはawait request.text()で生ボディを取る——request.json()で先にパースすると署名検証が壊れる罠に注意。

自分が試すなら

16の新規ならcacheComponents: trueを前提に、GETは「触れたデータで動的/静的が決まる」モデルで設計する。重い取得は本体に直書きせずuse cacheヘルパ（cacheLife('hours')）へ切り出す。内部変更の8割はServer Actions、実HTTP面が要る2割をRoute Handlerに——という配分が現実的だ。

注意点

よくある罠はparamsのawait忘れ、GETがキャッシュされない/されすぎる、Webhookでjson()して署名検証が壊れる、route.tsとpage.tsの同居コンフリクト。認可はproxy.ts（旧middleware）任せにせずRoute HandlerごとにverifySession()で再検証する（CVE-2025-29927）。

ひとことまとめ

「Reactから呼ぶ？→Yesは基本Server Actions、外部が叩く/HTTP制御が要る→Route Handler」。GETのdynamic既定とparamsのawaitだけは今すぐ頭を上書きしておく。
