useLinkStatus——「押した瞬間」に反応するナビゲーション・ペンディングUI（pending と loading.js の住み分け）

App Router には router.events が無く、リンクを押しても裏で処理中なのに無反応、という体感遅延が弱点だった。Next.js 15.3.0 で入り現行16.2.x の useLinkStatus は、クリックしたリンクが遷移完了するまでの pending 状態を返し、押した箇所の近くにインラインで即応を返す。派手なグローバルバーではなく、遅い動的ルートの離脱を減らすマイクロインタラクションだ。

📅 2026年7月24日 ⏱ 約3分 🔗 出典 3 件

背景。Pages Router 時代の router.events（routeChangeStart 等）は App Router に存在しない。だから NProgress 的なグローバルローディングが素直に配線できず、「リンクを押したのに何も起きない（実際は裏でプリフェッチ/レンダリング中）」という体感遅延が長年の弱点だった。useLinkStatus（Next.js 15.3.0 導入、現行16.2.x）はこの穴を「リンク単位」で塞ぐ公式フックだ。

何が新しいか（APIはこれだけ）

import { useLinkStatus } from 'next/link' → const { pending } = useLinkStatus()。引数なし、返りは { pending: boolean } のみ。pending は「URL が更新される前は true、更新後は false」。必ず <Link> の子孫の Client Component（'use client' 必須）で呼ぶ。

仕組み——2フェーズを分ける

ナビゲーションは2フェーズ。pending フェーズは遷移確定・URL更新の「前」（まだ現在ルートに居る）で、useLinkStatus が捉えるのはここ。loading フェーズは URL 更新の「後」〜新ルートの中身が出るまでで、ここは loading.js（ルートレベルの Suspense フォールバック）の担当。押下直後の即応は useLinkStatus、遷移後の骨組みは loading.js、と重ねて使う。

落とし穴

最大の罠はプリフェッチ済みだと pending が立たないこと。本番でリンクがビューポートに入りプリフェッチ完了していると遷移がほぼ即時になり、出したい時に出ない。だから効くのは prefetch={false} のリンクや、プリフェッチが間に合わない動的ルート。次にレイアウトシフト——pending 時だけ要素を出し入れすると CLS が出るので、固定サイズで常時レンダリングし opacity/visibility のトグルだけで見せ隠しする。速い遷移のチラつきは animation-delay: 100ms で「実際に時間を要したときだけ」フェードインさせる疑似デバウンスが定石。Pages Router は常に false、router.push 由来の遷移も対象外だ。

ひとことまとめ

公式の位置づけは "quick patch"。まずプリフェッチと loading.js で根本を速くし、それでも残る遅い遷移にだけ useLinkStatus を当てる。乱用してどこでもスピナーを出す設計は非推奨。グローバルなトップバーが要る場合は useRouter をラップして startTransition の isPending を使う自前フックで補う。
