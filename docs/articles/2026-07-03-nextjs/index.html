Payload CMS 3.x — Next.js に「直接インストール」して Server Component から DB を直クエリ

ヘッドレスCMSは「別サービスを立てて REST/GraphQL で叩く」が常識だった。Payload 3.0 は Express を捨て、Next.js の /app に丸ごと同居する初のCMS。管理画面もAPIもあなたのプロジェクト内に生成され、Local API で Server Component から HTTP を経由せず DB に直行できる。2025年に Figma が Payload チームを買収、v3.73.0 で Next.js 16 対応。ただし App Router 前提で、サーバー境界と認可の扱いに落とし穴がある。

なぜ今これが話題か

従来のヘッドレスCMSは、フロントとは別のバックエンドを立て、REST/GraphQL越しにコンテンツを取りに行く構成だった。デプロイ対象が増え、ネットワーク往復も挟まる。Payload 3.0(TypeScript製・MIT)はこの前提を壊す。Next.js の /app フォルダに直接インストールできる初のCMSとして、Express を捨て Next.js のランタイム内に丸ごと同居する。Admin Panel も API も全部あなたの Next.js プロジェクト内に生成され、デプロイは1つのまま。Vercel のサーバーレスにもそのまま載る。

何が新しいか

最大の武器が Local API。getPayload({ config }) で初期化済みインスタンスを取り、payload.find/create/update でHTTPを経由せず DB に直行する。これができる主要CMSは Payload だけで、React Server Components のデータ取得と極めて相性が良い。設定は payload.config.ts が単一の真実で、コレクション定義からTypeScript型・DBスキーマ・Admin UI・API がすべて生成される。DBアダプタは PostgreSQL/MongoDB/SQLite。エコシステム面ではFigma が Payload チームを買収(2025)、コアは OSS/self-host のまま維持され、v3.73.0(2026-01-23)で Next.js 16 対応済み。

仕組みと実装

既存プロジェクトへは npx create-payload-app@latest で導入でき、/app に (payload) ルートグループと payload.config.ts が入る。Server Component からは 'use client' 不要で、そのまま DB に直行できる。

// app/(app)/blog/page.tsx — Server Component
import 'server-only'
import { getPayload } from 'payload'
import config from '@payload-config'

export default async function BlogPage() {
  const payload = await getPayload({ config })
  const { docs } = await payload.find({

    注意点

    App Router 前提で Pages Router には噛み合わない。Local API はサーバー専用なので getPayload を含むモジュールをクライアントに引き込むとバンドルが壊れ秘密も漏れる——import 'server-only' でガードする。Server Action からの変更は既定で overrideAccess: true になりがちで、ユーザー入力起点では overrideAccess: false ＋ user を渡してアクセス制御を効かせないと認可を素通りする。Next.js のメジャー更新には Payload 側の対応バージョン(15→16 は 3.73)が要り、両者の整合を取る。

    ひとことまとめ

    Payload 3.x は「CMSを別サービスにしない」発想で、RSC のデータ取得を最短にする。1デプロイで完結し Local API が速いが、App Router 前提とサーバー境界・認可の作法を守ることが条件だ。
