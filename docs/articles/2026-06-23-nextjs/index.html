SVGの自己描画とモーフィング — stroke-dasharray と pathLength で純CSS、Server Component のまま置ける

ロゴが線で描かれていく self-drawing 演出や、アイコンが滑らかに別形へ変わるモーフィングは、SVGならテキストとして1要素ずつ制御でき、解像度非依存・軽量・アクセシブルに作れる。中心は stroke-dasharray / stroke-dashoffset と、長さを正規化する pathLength。純CSSで完結しGPU合成と相性がよいため、Next.js では 'use client' 不要で Server Component のまま置けるのが大きな利点だ。

なぜ今これが話題か

アニメーション手法の中で SVG は、ロゴの線が描かれていく「self-drawing（自己描画）」やアイコンのモーフィングを、解像度非依存・軽量・アクセシブルに実現できる。要素をテキストとして1つずつ制御でき、しかも純CSSで完結する。Next.js では純CSSアニメは JS ランタイム不要なので、'use client' を付けずに Server Component のまま置けるのが実装上うれしい。

何が新しい/重要か

self-drawing の原理はシンプルだ。stroke-dasharray を「線の全長 L」に設定すると実線 L と空白 L が交互になり、stroke-dashoffset: L で全長ぶんずらして非表示にできる。これを L → 0 へアニメーションすると線が描かれていく。問題は L がパスごとに違うこと。そこで pathLength="1"（または100）を付けると、ブラウザがそのパスを「指定した数値の長さ」として扱うので、形が違っても同じ数値でダッシュ計算できる。getTotalLength() の実測が不要になりコードが安定する。

モーフィングの選択肢

「アイコンA→B」の形状変化は手段で難度が違う。実用上は 2つのパスを重ねて opacity/transform でクロスさせる擬似モーフが軽く確実。d 属性そのものを補間する真のモーフィングは頂点数を揃える必要があり、素のCSS補間は不安定なのでライブラリ（GSAP/kute.js 等）が無難だ。

Next.js（App Router）での組み込み

純CSSの stroke アニメは SVGをJSXに直接インラインすれば Server Component のままでよい。別ファイルの .svg を動かしたいなら SVGR を使うが、Next.js 16 の Turbopack は webpack 設定を読まないため next.config.js の turbopack.rules 側に書く。

// next.config.js（Next.js 16 / Turbopack）
turbopack: { rules: { '*.svg': {
  loaders: [{ loader: '@svgr/webpack', options: { icon: true } }],
  as: '*.js',
} } }

自分が試すなら / 注意点

  • ロゴに pathLength="1" + @keyframes で描画演出を入れ、animation-fill-mode: forwards で完成形を保持。
  • scroll-driven animations の animation-timeline: view() と組み、見えたら描き始める。
transform/opacity はGPU合成で軽いが、fill/stroke/d のアニメはCPUのrepaintを伴い重い。stroke-dashoffset も描画に関わるためモバイルでは要計測。prefers-reduced-motion: reduce では完成形を即表示し、動きだけ消して情報は残す。

ひとことまとめ

「pathLength で正規化 → 純CSSで描画 → Server Component のまま」が今日の要点。まずはロゴ1つの self-drawing から入れると効果が分かりやすい。
