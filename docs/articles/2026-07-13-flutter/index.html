M3 ExpressiveはコアSDKに来ない — material_ui時代の幕開けとFlutter 3.44の現在地

umbrella issue #168813が明言：Material 3 Expressiveの新規開発はコアSDKでは行わず、分離後の新パッケージmaterial_ui / cupertino_ui側で進む。安定版はFlutter 3.44.0 / Dart 3.12.2。

📅 2026年7月13日 ⏱ 約3分 🔗 出典 4 件

GoogleがMaterial 3 Expressiveを発表して以来、「FlutterのコアSDKにいつ入るのか」は多くの開発者の関心事だった。その答えがumbrella issue #168813で明確になっている。2025年7月29日の方針決定により、コアSDKではM3 Expressiveを新規開発せず、contributionも受け付けない。material/cupertinoライブラリの分離後、すべての新規開発は新パッケージmaterial_ui / cupertino_ui側で行われる体制だ。

足元の安定版はFlutter 3.44.0 / Dart 3.12.2（次の安定版は2026年8月予定）。3.44ではImpellerがAndroid 10+で必須化され、GenUI（A2UIプロトコル）サポート、Agentic Hot Reload、MediaQueryからのdisplay corner radii取得が入った。Dart 3.12はprivate named parametersの正式化とprimary constructors（experimental）。ツールチェーンもflutter analyze --suggestionsがJava 25 / Gradle 9 / AGP 9 / Kotlin 2.2.20まで対応した。

アニメーション側の今日の深掘りはRiveランタイム。stableはrive 0.13.20で、Rive Nativeベースに書き直した次世代0.14.0-dev.6が開発中だ。state machineで描画とロジックを分離し、events（0.12+）でハプティクスやサウンドを正確にトリガーできる。ImpellerはAOTシェーダーコンパイルでジャンクを排除し複雑シーンの平均ラスタライズを約50%短縮しているが、Rive併用時に描画差が出る場合は--no-enable-impellerで切り分ける。状態管理はRiverpod 3.3.1 / bloc 9.2.xが現行で、基本線は変わらず「新規はRiverpod 3.x、大規模・監査案件はBloc」。動いているProvider/BLoCアプリはrepositoryを先に抽出して1機能ずつ移行すればよく、書き換えない勇気も実践則だ。

実務では、レンダリング4フェーズ（Build/Layout/Paint/Composite）を意識したconst徹底・不要ラッパー除去・RepaintBoundaryの戦略配置が基本動作。DevToolsのPerformance ViewでUI/Rasterスレッドのjank位置を特定してから手を付ける。注意点として、M3 Expressive対応を今のコアSDKで待ち続けても来ないので、新デザインの採用計画はmaterial_uiパッケージの動向を見て立てること。

ひとことまとめ: Materialの進化はコアSDKから新パッケージへ主戦場を移した。3.44の安定運用と並行して、material_uiのウォッチを習慣にしたい。
