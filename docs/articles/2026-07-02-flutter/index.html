Dart 3.12 の Macros 安定化 — build_runner なしでデータクラス/シリアライズをコンパイル時生成

Google I/O 2026 の本命は静的メタプログラミング。Dart 3.12 で Macros（マクロ）が安定化し、ライブラリがシリアライズ・データクラス・アダプタを『コンパイル時』に生成できるようになった。build_runner のような追加ビルドステップが不要になり、生成物とソースのズレやコード生成待ちから解放される。安定版は Flutter 3.44.0 / Dart 3.12.2。

📅 2026年7月2日 ⏱ 約3分 🔗 出典 3 件

なぜ今これが話題か

Flutter/Dart のデータクラスやJSONシリアライズは長らく build_runner によるコード生成頼みだった。freezed や json_serializable は強力だが、*.g.dart を生成する別ステップが要り、生成待ち・生成物とソースのズレ・CIの一手間がついて回った。Google I/O 2026 で示された Dart 3.12 の主役 Macros（マクロ）の安定化は、この構造そのものを置き換える。安定版は Flutter 3.44.0 / Dart 3.12.2で、次の安定版は2026年8月予定。

何が新しいか

Macros は静的メタプログラミング——コンパイラがソースを解析する過程で、ライブラリがコンパイル時にコードを生成できる仕組みだ。シリアライズ・データクラス・アダプタ生成がbuild_runner のような追加ビルドステップなしで完結する。生成物は中間ファイルとしてディスクに落ちず、コンパイルの一部として扱われるため、『生成待ち』も『.g.dart のコミット』も消える。あわせて pub workspaces が入り、モノレポで隣接する複数パッケージ（例：Flutter アプリと同居するサーバ関数）が依存を共有し、ワークスペース全体で一度に解決できる。

実務での使いどころ

DTO/モデル層のシリアライズやコピー・等価比較は、まさにマクロが効く領域だ。freezed のボイラープレートを更に削れる可能性があり、状態管理の推奨線（新規は Riverpod 3.3.1＋コード生成、監査/大規模は Bloc 9.x）とも噛み合う。ただし新規APIなので、まずは小さなモデル1つで生成物の挙動・ビルド時間・IDE補完を確かめてから面を広げるのが堅い。

注意点

Macros は強力だが安定化したばかり。既存の build_runner ベースの生成物と混在させると二重管理になりやすい。移行はモデル層から段階的に行い、生成物の差分・ビルド時間・実機での挙動を確認する。エコシステム（対応ライブラリ）の追随状況も見極めたい。Impeller 環境と同様、『新機能はまず小さく試す』が定石だ。

ひとことまとめ

Dart 3.12 は『生成の仕組み』を build_runner からコンパイル時マクロへ移す転換点。新規のデータクラス/シリアライズは、まず小さくマクロを試して感触を掴むのがよい。安定版 3.44.0 / 3.12.2 を土台に据えよう。
