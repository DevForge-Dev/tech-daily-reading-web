WebAssembly を Flutter Web の既定へ — dart compile wasm は stable、速度2〜3倍・ロード−40%・メモリ−30%

2026ロードマップ「Finishing the Job」は Wasm を Web の既定ターゲットに。dart compile wasm は stable 昇格済みで、JS比 速度2〜3倍・ロード約40%短縮・メモリ約30%削減。移行の鍵は dart:html/package:js を dart:js_interop へ置換し、flutter build web --wasm の dry-run で依存を棚卸しすること。

📅 2026年7月17日 ⏱ 約2分 🔗 出典 4 件

今日の主役は 「WebAssembly を Flutter Web の既定へ」。2026 ロードマップ「Finishing the Job」は Wasm を Web の既定ターゲットに据えることを掲げ、dart compile wasm はすでに stable 昇格済みだ。安定版は Flutter 3.44.0 / Dart 3.12.2。

何が変わるか（数値）

Dart→Wasm は JSターゲット比 2〜3倍高速。テストではロード時間 約40%短縮・実行時メモリ 約30%削減という結果が出ている。Wasm-GC が主要ブラウザのネイティブ機能となり、DevTools も WASM 既定で巨大ツリーを滑らかに描ける。

移行の前提

鍵はレガシーJS interop（dart:html / package:js）を Wasm-safe な dart:js_interop へ置換すること。まず flutter build web --wasm のdry-runを走らせ、非対応依存（dart:html 直接依存プラグイン等）を棚卸しするのが第一歩だ。

その他の実務ネタ

アニメは flutter_animate で .animate().fadeIn().slideY() のように Effect をチェーンし、onPlay:(c)=>c.repeat(reverse:true) でループも手組み不要。状態管理は @riverpod＋build_runner のコード生成が既定の書き方になり、テストは ProviderContainer(overrides:[...]) で UI を起動せず検証できる。リストは ListView.builder＋itemExtent でレイアウト計算を省き、120fps（フレーム予算8.3ms）を狙う。

注意点

Wasm は依存プラグインが足を引っ張るのが最大の壁。dart:html 直参照のプラグインは差し替えが要る。次の機能リリースは2026年8月予定なので、いま dry-run で棚卸ししておくと移行がスムーズだ。

ひとことまとめ

Web 配信する Flutter は「Wasm 前提」へ。--wasm の dry-run で棚卸しし、interop を dart:js_interop に寄せるのが今やること。
