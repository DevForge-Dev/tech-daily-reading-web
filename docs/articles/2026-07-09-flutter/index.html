フルスタック Dart へ — Cloud Functions for Firebase に experimental の Dart サポート

クライアントもサーバも同じ Dart で書ける時代の入口。HTTPS / callable 関数と Dart Admin SDK が初期提供され、Firebase CLI が手元でコンパイルして Cloud Run functions へ直接デプロイする。

📅 2026年7月9日 ⏱ 約3分 🔗 出典 4 件

Flutter 開発の長年の割り切りは「クライアントは Dart、バックエンドは TypeScript / Go / Python」だった。結果として2つのコードベースを保守し、ビジネスロジックとデータモデルを二重管理することになる。Cloud Next で発表された Cloud Functions for Firebase の experimental な Dart サポートは、この分断を解消する一歩で、Flutter / Dart / Firebase を通してクライアントもサーバも同一言語・同一ツールチェーンで書けるようになる。

初期リリースがサポートするのは HTTPS functions と callable functions。合わせて公開された experimental Dart Admin SDK は Cloud Functions 内で自動初期化されるため、追加設定なしで Firestore の読み書きができる。デプロイの仕組みも面白く、ソースを Cloud Build に上げるのではなく、Firebase CLI が開発マシン上で Dart をコンパイルし、生成バイナリを Cloud Run functions に直接アップロードする。AI エージェント基盤の Genkit Dart と組めば、AI ロジックもサーバ側 Dart に載せられる。

言語側の足場も揃ってきた。安定運用は Flutter 3.44.0 / Dart 3.12.2 で、Dart 3.12 は private named parameters（先頭アンダースコアの自動省略）や dot shorthands（.value 省略記法）に加え、primary constructors を experimental サポート。次の安定版は 2026年8月予定（年4回リリース）だ。クライアント側の今日の要点としては、jank 診断では setState で width / height を変えて再レイアウトを誘発するより Transform / Opacity の純グラフィカル変換へ寄せること、Impeller では Platform View / Texture が Skia より遅くなる報告（issue #180831）があるため該当画面は実機計測すること、の2つを覚えておきたい。

自分が試すなら、既存アプリの「クライアントから叩く小さな検証 API」を callable 関数1本で Dart 化し、Firestore の読み書きとデプロイ体験を確認するところから。モデル定義を共有パッケージに切り出せるかも見どころだ。注意点は experimental ステータスそのもので、予告なく変わる可能性があり、Firebase Console にまだ Dart functions が表示されない。本番ワークロードへの採用は時期尚早。

ひとことまとめ: 「Dart はクライアント言語」という前提が崩れ始めた。まずは callable 1本、遊びで踏むには十分な入口。
