App Router の国際化(i18n)——[locale]セグメント × proxy.ts × next-intl で静的化とhreflangまで通す

多言語化はURL設計・ロケール検出・静的化・SEO(hreflang)までを含む横断課題。前提として App Router には Pages Router の組み込みi18nが無く、next.config の i18n は効かない。2026年は next-intl が事実上の標準。Next.js 16 で middleware.ts が proxy.ts に改名され、next-intl v4 も追従済み。静的化には generateStaticParams＋setRequestLocale＋メタデータへの locale 明示の3点セットが要る。

この日のNext.jsはApp Router の国際化(i18n)。翻訳文字列の差し替えだけでなく、URL設計・ロケール検出・静的化・SEO(hreflang)までを含む横断的な設計課題で、App Router では特に落とし穴が多い。

最大の前提——組み込みi18nは無い

App Router には Pages Router のような組み込みi18nルーティングが無い。next.config の i18n（locales/defaultLocale）はPages Router 専用で App Router では効かない。公式も「動的セグメント [locale] ＋ 外部ライブラリ」で自前構成を推奨。2026年は next-intl が事実上の標準で、RSC前提で翻訳が素直に動く・1つのproxy設定でロケールネゴシエーション・全翻訳キーを型付けできる、が採用理由だ。もう一つ重要なのが、Next.js 16 で middleware.ts が proxy.ts に改名された点。next-intl v4 も createMiddleware を proxy.ts に置く構成へ更新済み。

静的化の落とし穴

locale-based routing では、Server Component で useTranslations を呼ぶと内部でヘッダ参照が走りルートが動的化する。静的化するには (1) generateStaticParams で対応ロケールを列挙、(2) setRequestLocale(locale) を全 layout / 全 page で next-intl API 呼出前に実行（Next.js は layout と page を独立レンダリングするため両方に必要）、(3) メタデータも静的化対象なので generateMetadata では getTranslations({locale, namespace}) に params の locale を明示、の3点が要る。setRequestLocale は React の cache() でリクエストスコープにロケールを書き、全 next-intl API がヘッダ代わりにそこから読む仕組みだ。

SEO(hreflang)と実務メモ

Metadata API の alternates.languages で hreflang link を生成し、x-default を必ず入れる（どのロケールにも一致しないユーザー向けフォールバック＝既定ロケール）。matcher で api/_next/_vercel/ドット付き静的ファイルを除外し忘れると、アセットにまでロケールプレフィックスが付く。useTranslations は RSC/Client どちらでも動き、RSCで直接使うのに 'use client' は不要（async な Server Component では getTranslations を await）。自分が試すなら、まず routing.ts に locales/defaultLocale を集約し、proxy.ts に createMiddleware(routing) を置く最小構成から始める。

ひとことまとめ

「組み込みi18nは無い→next-intl／middlewareはproxy.tsへ／静的化は3点セット＋hreflangにx-default」。この骨格を押さえれば App Router の多言語化は堅く通せる。
